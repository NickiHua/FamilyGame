# Dual-Grid 瓦片系统：核心逻辑与心智模型

> 这是一篇**知识/原理**文档（不是某天的开发日志）。目标：把 dual-grid 到底在干什么讲透，尤其是后来才真正想通的两条：
> 1. **「0.5 格偏移」是这套系统的灵魂**——不是画面 bug，是它必须存在的东西；
> 2. **要跟周围瓦拼得上，接缝必须落在「同一个像素位置」**——同一条边在相邻两张瓦上必须是像素级重合的。
>
> 项目里的实现：`Assets/Scripts/Editor/DualGridBuilder.cs`（铺图）、`scripts/tiles/gen_dual_tiles.py`（生成 16 张瓦）。踩坑实录见 `docs/developing_process/2026-07-01_process.md`。

---

## 0. 一句话原理

普通 autotile 是「**一格地形 = 一张瓦**」，得为每种邻接组合准备 47/48 张瓦（blob 集）。
Dual-grid 换了个角度：**显示瓦不画在格子里，而画在「4 个格子相交的角点」上**。每张显示瓦只看它四个角各是什么地形 → 4 个角 = 4 个 bit = **2⁴ = 16 张瓦**就够，接缝天然平滑。

代价就是：**显示网格相对地形网格整体错开半格**。这就是下面第 1 节要讲的「0.5 格」。

---

## 1. 灵魂：0.5 格偏移（错开半格）

有两张网格，**互相错开 (0.5, 0.5) 格**：

- **地形网格（terrain / data grid）**：逻辑数据。每个 cell 存一种地形（草/路/土/水…）。这是 `stage1_map.json` 里的东西，也是寻路/站位用的格子。
- **显示网格（dual / display grid）**：真正画出来的瓦。它的每个瓦**坐落在地形网格 4 个 cell 的公共角点**上，所以整体比地形网格**偏移半格、并且多出一圈**。

```
地形 cell 中心：  ●      ●      ●        (整数坐标 0,1,2…)
                    ┌──┐                  ← 显示瓦坐落在 4 个 ● 的中间
                    │▓▓│  = 一张 dual 瓦，覆盖 (x+0.5, y+0.5)
                    └──┘
地形 cell 中心：  ●      ●      ●
```

一张坐落在 `(x+0.5, y+0.5)` 的 dual 瓦，它的**四个角分别压在四个地形 cell 上**：

```
   角1 = terrain(x,   y+1)      角2 = terrain(x+1, y+1)
              ┌─────────┐
              │  dual    │
              │  瓦在此  │
              └─────────┘
   角4 = terrain(x,   y)        角8 = terrain(x+1, y)
```

把「这个角是不是目标地形」当成 bit：

```
bits = 1·[角1] + 2·[角2] + 4·[角4] + 8·[角8]      → 0..15，共 16 种
```

- `bits=0`：四角都不是该地形 → 空瓦（跳过）。
- `bits=15`：四角全是该地形 → 满瓦（纯色内部）。
- 中间 14 种 = 各种边/角/对角过渡。

> **项目里的 bit 约定（Unity y-up）**：`bit1=On(x,y+1)` 左上、`bit2=On(x+1,y+1)` 右上、`bit4=On(x,y)` 左下、`bit8=On(x+1,y)` 右下。PNG 左上角对应 bit1。

### 1.1 「错误的 0.5 tile」——最常见的翻车点

因为有这半格偏移，**任何一处忘了 / 多算 / 少算这 0.5 格，画面就会整体错半格**，症状通常是：

- 地形边缘看起来「往一个方向漂了半格」，或者水/路比数据里的位置偏了半个瓦；
- 边界瓦**探出地图矩形一圈**（因为显示网格本来就比数据网格多一圈，没处理就会戳出去）；
- 角色/物件按整数格站位，却发现脚下的地形瓦对不上——因为瓦是画在角点、不是画在格心。

**要记住的判断**：dual 瓦的坐标应该是 `(x+0.5, y+0.5)`（落在角点），**不是** `(x, y)`（落在格心）。看到「整体漂半格」先怀疑这里的 +0.5 有没有对。

外圈那一圈也要专门处理（要么裁掉、要么裁半格），否则半格 overhang 会戳出地图。见第 4 节。

## 2. 灵魂：接缝必须落在「同一像素位置」

这是拼图能不能拼上的**充要条件**，也是后来才真正想清楚的：

> **同一条边，在相邻的两张瓦上，必须是同一个像素位置（镜像重合）。**
> 只要两张瓦「共享的那条边」的过渡曲线不是逐像素对齐的，接缝处就会错位、露缝或叠边。

### 2.1 为什么

相邻两张瓦是**并排贴**的：左瓦的右边缘紧挨右瓦的左边缘。如果左瓦在「右边缘」把地形画到了 y=40px 高，右瓦在「左边缘」却画到 y=38px 高，那两张瓦一拼，接缝处地形高度**差了 2px**，就是一道裂缝或一个台阶。

所以规则是：

- **所有 16 张瓦，凡是带某一条边（比如「上边是草」）的，这条边的轮廓必须逐像素一模一样。**
- 一张瓦的「右边缘剖面」必须 = 右邻瓦「左边缘剖面」的镜像。上/下、左/右同理。
- 换句话说：边缘轮廓只是**四个角的状态**的函数，**起点/终点像素、过渡形状都被这条边两端的角唯一决定**，不能这张瓦随手画高一点、那张随手画矮一点。

### 2.2 实操：从「同一套 mask 旋转」得来

保证像素对齐最稳的做法：**不要一张张手画 16 张**，而是画少数几张「原型 mask」，**用旋转/镜像生成其余的**。这样同一条边天然来自同一段像素，对齐是白送的。

项目里 `gen_dual_tiles.py` 就是这么干的：用户只画了 3 张手绘 mask，脚本旋转出 16 张：

- `corner`（1 个角）→ 旋转出 bits 1/2/4/8；
- `half`（半张，一条直边）→ 旋转出 bits 3/5/10/12；
- `three`（3 个角，缺一角）→ 旋转出 bits 7/11/13/14；
- 满/空（15/0）单独处理。

> **反例（真踩过）**：`three` 的旋转映射把 **b11 和 b13 写反了**（PIL `ROTATE_90` 是逆时针），导致同一个水面凸起「顶角圆、底角是直角」。因为顶角用了对的瓦、底角用了缺口方向错的瓦——**两张瓦本该共享的边，起点像素对不上了**。
> **诊断法**：肉眼看不准就**量每张瓦四象限的不透明像素占比**，跟「这张 bits 应该实心哪几个角」对照，装反的瓦立刻现形。相似结构渲染不一致（顶圆底方）＝ 一定有 bug，别用「几何本来如此」搪塞。

### 2.3 无缝的「越线 delta」也要全局一致

过渡不是恰好停在瓦的中线，而是**越过中线探入邻侧一点点**（delta），这样地形边界才有机、不是直线。关键：**这个 delta 在所有 16 张瓦里必须是同一个值**。

- 本项目：草在每条混合边都伸到 **42px**（瓦 64px，中线 32px，越线 delta=10px）。
- 只要每张瓦的「越线量」一致，任意两张瓦相接时**边界穿过接缝的位置就是连续的**，就无缝。
- 想加粗过渡又不想重画 mask：可以对每张瓦的不透明区域**统一膨胀（PIL MaxFilter）**——因为是统一操作，所有接缝一起移动，仍然对齐、仍然无缝。

## 3. 多地形：优先级叠层（本项目的做法）

标准 dual-grid 讲的是「一种地形 vs 空」。要在一张图上放草/路/土/水多种，本项目用**优先级压盖**：

- 每种地形**各自跑一遍 dual**（共用同一套 16 瓦形状，只是填不同纹理），按优先级**从低到高叠画**。
- `On(角) = Priority(该角地形) >= 本层优先级`。高优先地形的 dual 层画在后面/上面，**往低优先地形方向探 delta 半格**，形成「高地形啃进低地形」的有机边。
- 各层用**完全相同的瓦形**，所以高层正好盖住低层、不留缝。

**优先级顺序（踩坑后定的）**：`水/桥 0 < 路 1 < 土 2 < 草 3`。

> **为什么路要放最低**：优先级压盖 hybrid 有个陷阱——**一条直边，在「邻居的相对优先级发生翻转」的地方会跳 2×delta**。竖直的路，如果上段邻居是更高优先的草（草吃路，路缘 +11）、下段邻居是更低优先的土（路吃土，路缘 −11），路缘就在草/土交界处**硬跳 22px**（直角台阶）。
> 把「穿场型」地形（路）压到**最低**，它永远被吃、边缘方向恒定，就不跳了。通则：**让贯穿全图的线状地形（路/河）当最低优先级**。

- **水**：不做 dual 层，只做 base（见下）。水边由更高地形的 dual 压过来形成。
- **桥**：`'D'` 格作为**独立的最上层 object**单铺，角色再 Y-sort 在其上。

## 4. 边界处理：外圈那半格怎么办

显示网格比数据网格**多一圈**（因为瓦画在角点上）。这一圈会探出地图矩形，必须处理：

1. **全外圈循环 + clamp**：铺图时循环 `for y=-1..h-1, x=-1..w-1`，出界的采样 **clamp 到最近边缘格**。好处：纯色边采样全同 → `bits=0/15` 被跳过（不冒假地形）；只有边界上真有地形变化的地方才 blend。
   - 注意：**不能简单用「内圈 only」跳过外圈**——那样地图边缘的地形变化会变成硬直切、不 blend。
2. **裁边瓦（clipped border tiles）**：clamp 后外圈只会用到竖边 `3/12`、横边 `5/10`。给每种地形额外生成 **8 张「裁掉外半格」的变体**：`03L 03R 12L 12R`（左右边）+ `05T 05B 10T 10B`（上下边）。铺外圈时按边选变体，把探出地图的那半格抹掉。
   - **优点**：裁剪**烘进 PNG 资源**，所见即所得、可离线验证；**不要**用运行时 `SpriteMask` 之类去裁 Tilemap（本项目里对 Tilemap 没生效，且无法本地验证）。

### 4.1 Y 朝向大坑：验证图 y-down vs Unity y-up

做边界裁剪时被这个坑翻过车，务必记住：

- **Python/Pillow 验证渲染是 y 向下**（JSON row0 在图顶）；
- **Unity Tilemap 是 y 向上**（cell y=0 在屏幕底，经 `TerrainAt(x,y)=base[H-1-y][x]` 翻转）。
- **内部瓦形两边看着一致**（都是「地图西北角 → PNG 左上角」）；
- 但**任何「循环下标 ↔ 屏幕哪条边」的逻辑会上下镜像**！比如 `y==-1` 这一圈：在 Python 验证图里是**顶边**，在 Unity 里是**底边**。
- 所以裁边选向在 Unity 里必须是 `y==-1 → "B"`（底）、`y==h-1 → "T"`（顶），跟 Python 直觉相反。X 不受影响（左右两边一致）。

> 教训：做任何「边缘/循环下标」相关逻辑前，先在纸上把 y-down/y-up 这层翻转想清楚，别拿 Python 验证图的方向直接照搬进 Unity。

## 5. 外部生成资源要 force-import

脚本外部写出来的 PNG（比如那 8 张裁边瓦），**Unity 不会自动导入**。铺图时 `GetOrCreateTile` 若 `LoadAssetAtPath` 拿不到、但文件确实存在，要 `AssetDatabase.ImportAsset(path, ImportAssetOptions.ForceSynchronousImport)` 强制导入，否则会**回退成整张瓦、裁剪失效**（尾巴又戳出来）。加载仍失败就 `LogWarning` 报出缺哪张瓦，别静默回退。

## 6. 速查清单（Checklist）

铺 dual-grid 时逐条自检：

- [ ] dual 瓦坐标是 `(x+0.5, y+0.5)`（角点），不是 `(x,y)`（格心）——**别漏那 0.5 格**。
- [ ] 16 张瓦最好由**少数 mask 旋转/镜像生成**，保证同一条边逐像素对齐。
- [ ] 每张瓦的**越线 delta 全局一致**（本项目 42px / delta 10）。
- [ ] bit 约定固定：`1=左上 2=右上 4=左下 8=右下`，PNG 左上=bit1。
- [ ] 多地形用优先级叠层；**线状地形（路/河）放最低优先级**避免台阶。
- [ ] 外圈用**全圈循环 + clamp**，再用**裁边瓦**抹掉探出的半格（别用运行时遮罩）。
- [ ] 边界裁向记得 **Unity y-up**：`y==-1→B`、`y==h-1→T`。
- [ ] 外部生成的瓦 PNG 在 builder 里 **force-import** 兜底。
- [ ] 验证：相似结构必须渲染一致；不一致就**量四象限 alpha 占比**定位装反的瓦。
