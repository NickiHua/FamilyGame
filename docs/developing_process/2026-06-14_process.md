# 2026-06-14 开发日志：demo 地图逻辑网格逐格人工校对（JSON ↔ AI 成图对齐）

## 一、当日目标

承接 06-13 的成果：已经有了 AI 成品图 `art/maps/demo_map.png`（2240×2240）和它的逻辑网格 `demo_map.json`（35×35）。

但 06-13 末尾已知一个隐患：**成品图的布局和 JSON 不完全一致**。原因是出图时 ControlNet 给了模型一定自由度，模型把河、桥、房子、悬崖画得和 JSON 原始布局有偏差（河更宽、整体右移、桥的行数错位、右岸那片悬崖 C 大面积画成了水和草）。

所以今天的目标只有一个：**让 `demo_map.json` 的每一格地形字母，精确对上 AI 成图里实际画出来的东西**。这样 Unity 里的 walkable 判定（红色阻挡块）才会和玩家看到的背景图严丝合缝——不会出现"站到水里"或"明明是草地却走不过去"。

地形字母约定（本图调色板）：
- `G` 草地、`R` 道路、`D` 桥 → **可走**
- `W` 水、`C` 悬崖/礁石、`B` 房屋、`L` 墙、`F` 森林 → **阻挡**（F 是当天最后才加进阻挡的，见第六节）

## 二、第一次尝试：自动取色分类器（失败/不够用）

最直接的想法：**让程序自动**比对成图和 JSON，不靠人眼。

管线里本来就有一个 `scripts/map/sample_terrain.py`：把成图按 35×35 切格，取每格中心 50% 区域的平均 RGB，再用一套启发式阈值（蓝=水、低饱和度灰=悬崖、暗绿=森林、红橙=房顶、tan=路、绿=草）猜每格地形，和当前 JSON 比对，打印差异。

**跑出来的结果**：1225 格里 **580 格不一致**。

**为什么这个数字不能用**：绝大多数是**分类器的噪声**，不是真错：
- HD-2D 的 tilt-shift 暗化 / 阴影，把大片**草地 G 读成了森林 F**。
- 桥的**木色**、暗色屋顶被当成 `B`。
- 礁石和阴影边界互相串色。

也就是说，自动分类器对"颜色明确"的水 W 还算准，但对草/林/路/桥这种**色相接近、又被光影污染**的地块完全不可信。结论：**纯自动不行，必须人眼最终裁决**。

但人眼也有个大问题——见下一节。

## 三、第二次尝试：整图缩略图人工看（失败，看不清）

让我（AI）直接看整张 35×35 的叠加图判断。

**失败原因**：35×35 缩略图里每个格子太小，叠加上字母和格线后更糊，**我根本分不清某一格到底是深绿草还是森林、是水边还是礁石**。而且我（AI）每次"看图"会消耗图片预算，反复看整张图很快就把预算耗尽，越看越没法看。

**教训**：地图校对这种活，"看整张大图"是最差的方式——既看不清又费预算。

## 四、最终方案：7×7 分块 + tracking JSON（成功）

用户提出了关键的好点子：**把整图切成 7×7 = 49 块**，35÷7 正好每块 5×5 格。命名 `R1-C1`…`R7-C7`（R=块行从上到下，C=块列从左到右）。用户负责一块块看、报哪些格要改；我负责改 JSON + 记录进度。

### 4.1 写了 `scripts/map/slice_blocks.py`

干什么：
- 把成图切成 49 个 5×5 块，每块**放大到 768px**（关键——放大后才看得清地形）。
- 每块叠加：① 每格当前 JSON 字母（黄=阻挡，白=可走）；② **每格的全局坐标 `列,行`**（左上角青色小字）。坐标是为了让用户能精确指"(18,21) 改成 W"，我能直接定位 `base[行][列]`。
- 同时生成/合并一份 **tracking JSON** `scripts/map/out/review_blocks.json`：49 个块，每块有 `cols`/`rows` 全局范围 + `status`（pending/reviewed/fixed/ok）+ `notes`。**重跑时保留已有状态**，所以能 track "哪些块我们一起过过了"。

块覆盖范围速查：
- 列：C1=0–4, C2=5–9, C3=10–14, C4=15–19, C5=20–24, C6=25–29, C7=30–34
- 行：R1=0–4, R2=5–9, R3=10–14, R4=15–19, R5=20–24, R6=25–29, R7=30–34

### 4.2 每改一块的固定流程（每行重复）

1. 用户看 `scripts/map/out/blocks/R?-C?.png`，报"哪些格改成什么"。
2. 我 `read_file` 读出 `demo_map.json` 里对应那几行的**精确原文**（绝不靠记忆，因为前面行改过后行内容会变）。
3. 逐格计算新行，`replace_string_in_file` 改 `Assets/Art/Maps/demo_map.json`（这是 Unity 和渲染都读的真值源）。
4. 更新 `review_blocks.json` 对应块的 status + notes（把用户的原始指令记进 notes，方便回溯）。
5. 重跑同步+渲染三连：
   ```
   Copy-Item Assets/Art/Maps/demo_map.json scripts/map/demo_map.json -Force
   python scripts/map/overlay_blocked.py
   python scripts/map/slice_blocks.py
   ```
   即：同步副本 → 重画整图叠加 → 重切 49 块（用新字母），供下一轮复核。

就这样一大行一大行（R1→R7）推进，把整张图过了一遍，又做了多轮零散微调。

## 五、过程中踩的坑（重要，越细越好）

### 5.1 坑：我把"纯森林行"当背景跳过，实际有村庄
最初我以为外圈一大圈是纯 F、可以直接标 ok。改 R1 时读文件才发现 **row4 本来就是村庄行（含 B 房屋）**，不是纯森林——我之前的行编号有偏差。
**解决**：从此每改一行**必先 read_file 读真实原文**再算，不靠脑补的行内容。

### 5.2 坑：用户指令里有歧义 / 截断坐标
R5-C3 用户写"`10，,2 W`"——双逗号、行号被截断，无法确定指哪一格（col10 在 rows 20–24 当时全是 G）。
**解决**：**不猜**。把该块标成 `reviewed` 而非 `fixed`，notes 里明确写"PENDING: 歧义未应用"，并在回复里单独问用户。用户后来澄清是 `(10,20)=W`，才补上、转 `fixed`。

### 5.3 坑（当天最大的坑）：base 行长度不是 35，MapGrid 静默丢行
改 R7 时我手敲新行，**row33、row34 各多打了 1 个字符（36 个）**。

后果非常隐蔽：
- `overlay_blocked.py` 渲染照常出图（它按 35 等分像素，多 1 字符不影响出图），所以**叠加图看起来正常**，我没立刻发现。
- 但 Unity 的 `MapGrid.cs` 解析逻辑是：`Regex.Matches(text, "\"([GRFWCBLD]+)\"")`，只收**长度正好等于 grid_size(35)** 的引号字符串当作 base 行。那 2 行长度=36，被**直接丢弃**，`_rows` 只剩 33 行。
- `_rows.Length(33) != Size(35)` → 打了个 **Console warning**：`[MapGrid] Parsed 33 rows but grid_size is 35.`
- 更要命：解析不完整 → **`OnDrawGizmos` 不画红块**。表现为"Game 视图能看到红块、Scene 视图看不到"……（其实那会儿是被一连串无关的 UI 误操作掩盖，见 5.4）。

**怎么定位的**：用户贴出 Console 的那条 warning。我立刻写了一行校验脚本：
```
python -c "import json; b=json.load(open('Assets/Art/Maps/demo_map.json'))['base']; print([(i,len(r)) for i,r in enumerate(b) if len(r)!=35])"
```
直接打印出 `[(33,36),(34,36)]`——锁定就是这两行多了 1 字符。砍掉多余的 F 改回 35，warning 消失、红块恢复。

**沉淀为铁律**：**`base` 每一行必须严格 35 字符**。此后每次改完都跑这条 `bad rows` 校验，确认 `[]` 才渲染。这是当天最值钱的一条经验。

### 5.4 坑：一连串 Unity Gizmos UI 误操作（和代码无关，但耗了很久）
"Scene 看不到红块"排查时，用户连续点错了好几个 Scene 工具栏按钮，画面一度全灰：
- 先是 **Game 和 Scene 的 Gizmos 开关是独立的**——用户开了 Game 的，Scene 的没开。
- 然后误点了 **Scene 渲染模式下拉**，切到了"Lighting Visualization（灯光可视化）"模式，整个画面变灰只剩网格和 GI 图例。**解决**：渲染模式切回 `Shaded`。
- 又展开了 **Gizmos 类型下拉列表**，以为要在里面找 MapGrid 的勾（其实那个勾是开的）。
- 真正的总开关是 **Gizmos 按钮本体**（用户最后自己发现"这个球本身就是 Gizmos！"）。

**教训**：Scene/Game 的 Gizmos 是两个独立总开关；渲染模式下拉和 Gizmos 下拉是两个不同的东西，别混。不过这一段本质是被 5.3 的真 bug 误导——代码已经因为丢行不画了，UI 怎么点都没用，**先看 Console warning 才是正道**。

### 5.5 给 MapGrid 加了 Reparse 菜单
`OnDrawGizmos` 用缓存的 `_rows`，改完 JSON 不会自动重解析。为省去"进退 Play 才刷新"的麻烦，给 `Parse()` 加了 `[ContextMenu("Reparse JSON (refresh gizmos)")]`——改完 JSON 在组件 ⋮ 菜单点一下即可刷新红块。

## 六、收尾改动：森林 F 全部改为不可通过

校对接近尾声时，用户决定**森林 F 一律不可走**（之前是 walkable、move_cost=2）。这是个跨多文件的一致性改动，必须 4+ 处同步否则美术/逻辑/工具会漂移：

1. `Assets/Scripts/View/MapGrid.cs`：`Walkable = "GRFD"` → **`"GRD"`**（去掉 F），注释同步。
2. `Assets/Art/Maps/demo_map.json` 调色板：F `walkable:false, move_cost:0`。
3. `scripts/map/demo_map.json`：同步副本。
4. `scripts/map/build_demo_map.py` 的 `PALETTE`：F 同改（保证将来重生成 JSON 不回退）。
5. `scripts/map/overlay_blocked.py`：`WALKABLE=set("GRD")`，并给 F 加深绿 tint，让叠加图里森林也显示为阻挡。

改完可走地形只剩 **G / R / D**，其余全阻挡。

## 八、可复用的方法论（给未来的自己）

1. **大图校对 = 分块放大 + 坐标叠加 + tracking JSON**。别看整张缩略图（看不清又费图片预算），别指望纯自动分类器（光影污染导致一半假错）。
2. **改格子数据前必 read_file 读真实原文**，因为前面的改动会改变后续行内容，靠记忆必错。
3. **每次改完跑行长度校验**（`bad rows` 必须为 `[]`）。固定宽度网格里多/少一个字符会被解析器静默丢弃，bug 极隐蔽。
4. **症状不对先看 Console**：MapGrid 的解析 warning 直接指向了根因，比在 UI 里瞎点高效得多。
5. **歧义不猜**：标 reviewed + notes 写明 pending，回头问清再改。
6. **跨文件一致性改动**（如改地形可走性）要列清单逐处同步：运行时代码 / 数据 JSON+副本 / 生成器 / 可视化工具。
