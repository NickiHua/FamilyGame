## 三、工程侧：写了哪些脚本，每个干什么用（重点，给学习用）

今天 agent 在 `Assets/Scripts/` 下新建了 6 个 C# 脚本。下面逐个讲**它是干什么的、为什么这么写**。

### 3.1 运行时脚本（挂在角色 GameObject 上，游戏运行时工作）

放在 `Assets/Scripts/View/`，属于 spec §3.2 的「View 表现层」。

#### `Facing.cs` —— 方向枚举
最简单的一个文件，定义了角色的 4 个朝向：`South / East / North / West`。

为什么需要它：游戏逻辑里到处要表达"角色面朝哪"，用枚举（而不是字符串 "south" 或数字 0/1/2/3）能让代码更清晰、不容易写错。

关键约定：**West（朝左）不单独存美术**，运行时用 East 的图 + 水平翻转（`flipX`）来实现。这是 spec §5.2 定的省素材策略。

#### `CharacterSpriteAnimator.cs` —— 逐帧动画播放器（今天的核心）
这是替代 Unity 自带 Animator 的**自定义动画播放组件**。

**它干什么**：
- 内部存了一个"动画库"——4 个动作（Idle/Walk/Attack/Hit）× 3 方向（S/E/N）= 12 组帧序列（spec 里叫 Clip）
- 每帧根据"当前朝向 + 当前是否在移动"自动挑选该播哪组动画
- 按设定的 fps（idle 8 帧、walk 10 帧、attack 12 帧）逐帧切换 `SpriteRenderer.sprite`
- 朝左时自动设 `flipX = true` 复用朝右的图
- 提供 `PlayOneShot(Attack/Hit)` 播放一次性动作（攻击/受击不循环）

**为什么不用 Unity 自带的 Animator？**
Unity 标准做法是为每个动画手动创建 `.anim` 文件 + `.controller` 状态机（要在 GUI 里一个个拖、连线，很繁琐）。我们改成代码播放，好处是**整套角色装配可以用脚本自动生成**，不用手点 GUI。这是一个有意为之、可逆的偏离 spec §2 的决定——以后想换回 Animator，不影响其他代码。

#### `GridMover.cs` —— 网格移动组件
**它干什么**：让角色"一格一格"地在网格上移动，而不是瞬移或自由滑动。

- 你给它一个目标格子（或一串格子组成的路径），它会**平滑地从当前格滑到下一格**（用 `Vector3.Lerp` 插值，看起来是"走过去"）
- 移动时自动告诉 `CharacterSpriteAnimator`：①现在该朝哪个方向 ②现在是"走路"状态（播 walk 动画）
- 到达终点后触发 `Arrived` 事件
- `tilesPerSecond` 控制移动速度，`cellSize` 是一格多大（=1，因为 1 个 Unity 单位 = 1 格）

这就是 spec §5.4 要求的"角色移动是'走过去'，不是'瞬移'"。

#### `CharacterDemoController.cs` —— 键盘测试驱动（临时用）
**它干什么**：纯粹是阶段 1 用来**手动测试**的小工具，让你能用键盘操控角色验证动画。

- **方向键 ↑↓←→**：走一格
- **空格**：播攻击动画
- **H 键**：播受击动画

它用的是项目配置的新版 Input System（`Keyboard.current`）。等阶段 3 做真正的战斗输入时，这个会被删掉/换掉。它只是个"遥控器"，不是正式玩法。

### 3.2 编辑器脚本（只在 Unity 编辑器里工作，不进游戏）

放在 `Assets/Scripts/Editor/`。**Unity 规定：放在叫 `Editor` 的文件夹里的脚本只在编辑器里运行**，用来做工具、自动化，不会被打进最终游戏。

#### `PixelArtImportPostprocessor.cs` —— 导入设置自动守卫
**它干什么**：每当有 PNG 图片被导入到 `Assets/Art/Characters/` 或 `Assets/Art/Tilesets/` 下，它**自动**把导入设置改成像素游戏该有的样子，省得你每张图手动改。

它自动设置的项（spec §5.6 规定）：
- **角色**（Characters 文件夹）：Sprite 模式 Single、**PPU=48**、Point 滤镜、不压缩、Clamp
- **地块**（Tilesets 文件夹）：Sprite 模式 **Multiple**（为切片准备）、**PPU=16**、Point 滤镜、不压缩、Clamp

为什么重要：**最关键的是 Filter Mode = Point**。不改的话 Unity 默认用 Bilinear（双线性过滤），像素图会被"抹糊"成马赛克。这个脚本保证永远不会忘记改。

> 菜单里还有 **Tools → FantacyCentry → Reapply Pixel Art Import Settings**，对已经导入过的图强制重新套用设置（今天 tileset 复制进来时脚本还没编译、漏掉了，就是用这个菜单补救的）。

#### `CharacterRigBuilder.cs` —— 一键生成角色 prefab（今天最省事的工具）
**它干什么**：菜单点一下 **Tools → FantacyCentry → Build LingShuang Rig**，它就自动：

1. 扫描 `Assets/Art/Characters/LingShuang/animations/` 下的所有动画文件夹
2. 按文件夹名识别这是哪个动作（含 walk/idle/thrust/lance/attack/hit/flinch 等关键词）
3. 加载每个方向（south/east/north）下的 `frame_001`~`frame_008`（**自动跳过 frame_000**）
4. 组装成 12 组 Clip，喂给 `CharacterSpriteAnimator`
5. 创建一个带 `SpriteRenderer + CharacterSpriteAnimator + GridMover + CharacterDemoController` 的 GameObject
6. 存成 `LingShuang.prefab`

今天 Console 打印 `Built LingShuang rig with 12 clips` 就是它干的活。

## 四、如果不用脚本，纯手动在 Unity 里怎么做？（学习向，详细版）

用户提到第一次搞 LuLi 是手动的、现在记不清了。这里把**手动等价流程**完整记下来，对照理解今天脚本替你省掉了什么。

### 4.1 手动做角色动画 prefab（替代 `CharacterRigBuilder`）

**前提**：图已经导入 Unity 且导入设置正确（PPU、Point 滤镜等，见 4.3）。

**第 1 步：为每个动画方向创建 AnimationClip**
1. Project 面板进到某个动画方向文件夹，例如
   `Assets/Art/Characters/LingShuang/animations/<idle 动画>/south/`
2. 框选 `frame_001` ~ `frame_008` 这 8 张（**按住 Ctrl 点 frame_001，再 Shift 点 frame_008**；**不要选 frame_000**）
3. **整体拖到 Scene 视图里**
4. Unity 弹出 "Create New Animation" 对话框 → 命名 `LingShuang_Idle_South.anim` → 保存
5. Unity 会自动做三件事：①创建一个带 SpriteRenderer + Animator 的 GameObject ②创建这个 `.anim` 动画 ③创建一个同名 `.controller`（动画状态机）

**第 2 步：把其余 11 组动画也做成 Clip**
重复第 1 步，对 walk/attack/hit × south/east/north 共 12 组都做一遍（W 方向不做，靠翻转）。
命名约定：`{角色}_{动作}_{方向}`，如 `LingShuang_Walk_East`。

**第 3 步：调每个动画的播放速度**
1. 双击某个 `.anim` → 打开 Animation 窗口
2. 改 **Samples** 字段（就是 fps）：Idle 用 8，Walk 用 10-12，Attack 用 12-15
3. 每个 clip 都要调一次

**第 4 步：在 Animator 里连状态机**
1. 双击 `.controller` → 打开 Animator 窗口
2. 把 12 个 clip 拖进去成为 State
3. 创建 Parameters（如 `Action`、`Direction` 或 bool `IsMoving`）
4. 在 State 之间连 Transition（过渡箭头），设置条件（什么时候从 Idle 切到 Walk……）
5. **这一步极其繁琐**——12 个状态两两之间可能要连几十条线并设条件

**第 5 步：West 方向翻转**
给朝西时的逻辑设 `SpriteRenderer.flipX = true` 复用 East（需要额外写脚本或在 Animator 里处理）。

**第 6 步：存成 prefab**
把搭好的 GameObject 从 Hierarchy 拖回 Project 面板，存成 `.prefab`。

> **对比**：上面 6 步、十几个 clip、几十条状态机连线，今天 `CharacterRigBuilder` **一个菜单点击全包了**（代价是改用代码播放器而非 Animator，见 3.1）。这就是为什么今天这么快。

### 4.2 手动配 Pixel Perfect 相机（今天是手动做的，无脚本）

1. Hierarchy 选中 **Main Camera**
2. Inspector 最下面 **Add Component** → 搜 `Pixel Perfect` → 选 **Pixel Perfect Camera**
3. 填参数：
   - **Assets Pixels Per Unit = 48**
   - **Reference Resolution = 480 × 270**
   - **Grid Snapping = Pixel Snapping**（新版 Unity 6 把旧的 "Upscale Render Texture / Pixel Snapping" 合并成这一个下拉了）
   - Crop Frame = None
4. 看 **Current Pixel Ratio** 显示 `5:1` 表示生效（把 480×270 放大 5 倍铺满窗口）

作用：游戏内部以 480×270 低分辨率渲染再整数倍放大，得到**锐利、不抖动**的像素画面。

### 4.3 手动改像素图导入设置（替代 `PixelArtImportPostprocessor`）

每张 PNG 选中后，在 Inspector 改（spec §5.6）：

| 设置 | 改成 | 为什么 |
|---|---|---|
| Texture Type | Sprite (2D and UI) | 当 sprite 用 |
| Sprite Mode | 角色 Single / 地块 Multiple | 地块要切片所以 Multiple |
| Pixels Per Unit | 角色 48 / 地块 16 | 决定 1 单位 = 多少像素 |
| **Filter Mode** | **Point (no filter)** | **最关键，不改会糊** |
| Compression | None | 防颜色失真 |
| Wrap Mode | Clamp | 防边缘伪影 |

改完点 **Apply**。一张张改很烦——这正是写 `PixelArtImportPostprocessor` 自动化的原因。

```

```
团队其他人需要时各自从 Asset Store 免费下载。

### 5.3 尺寸决策：tile 16px vs 角色 48px

| 项 | 角色 (LingShuang) | tile 包 |
|---|---|---|
| 单位像素 | 48px | **16px** |

两者不同尺寸。**决策**：tile 用 **PPU=16**，让 1 个 16px 地砖 = 1 个 Unity 单位 = 1 格；角色 PPU=48 不变。

结果：角色 sprite 比单个格子大（占视觉上约 3 格高），但逻辑上只占 1 格——**正好还原 FFT / Tactics Ogre 那种"人比格子大"的战棋观感**（spec §5.1 本就这么定）。

## 六、手动铺 Tilemap 完整流程（今天走的，详细记录）

这是 Unity 战棋/2D 游戏的核心技能，完整记下来。

**第 1 步：切片地块图（Sprite Editor）**
1. Project 选中 `TX Tileset Grass`（确认导入设置 PPU=16、Point、Multiple）
2. Inspector 点 **Open Sprite Editor**
3. 窗口左上 **Slice** 下拉 → Type 选 **Grid By Cell Size** → Pixel Size **X=16, Y=16** → 点 **Slice**
4. 顶部 **Apply** → 关窗口
5. Project 里展开 `TX Tileset Grass`，会看到切出的 **256 个**小地砖 `TX Tileset Grass_0..255`（母图 256×256，按 16×16 切正好 16×16=256）

**第 2 步：打开 Tile Palette**
菜单 **Window → 2D → Tile Palette**

**第 3 步：新建调色板**
1. 面板左上 **Create New Palette** → Name `GrassPalette` → Grid = Rectangle → Create
2. 保存到 `Assets/Art/Tilesets`

**第 4 步：把地砖灌进调色板**
1. 把母图 `TX Tileset Grass` 整个拖进 Tile Palette 空白区
2. 选 Tiles 的保存文件夹（如 `GrassTiles`）→ 256 个砖铺满调色板（生成 256 个 `.asset`）

**第 5 步：创建 Tilemap（场景里的"画布"）**
1. Tile Palette 上方 **Create New Tilemap → Rectangular**
2. Hierarchy 自动出现 `Grid` → 下挂 `Tilemap`

**第 6 步：刷地图**
1. 在调色板里点选一块纯草砖
2. 选画笔工具（🖌）
3. 在 **Scene 视图**（不是 Game）按住左键拖动 → 刷出草地
4. 刷约 10×10 一片当测试地图

**图层顺序**：今天角色没被地图盖住，因为 `CharacterRigBuilder` 建 prefab 时把 SpriteRenderer 的 **Order in Layer 设成了 10**，高于 Tilemap 默认的 0，所以角色永远画在地块之上。手动做的话需要自己调这个值，或用 Sorting Layer。

### 6.1 「切割」到底改了什么 / 数据存哪（实测验证）

昨天在 Sprite Editor 里把母图切成 16×16，**没有产生任何新的 png，也没动 png 的像素**。切割信息（每个小块在大图上的矩形坐标）被写进了**源图的 `.meta` 文件**：

- `pixel-art-top-down/TX Tileset Grass.png.meta` → 142 KB / 6023 行，里面有 **256 个 sprite 矩形**（`name:` 256 条、`rect:` 256 条）。普通未切图的 meta 只有几十行。
- 我们做的 256 个 `GrassTiles/*.asset` 是 **Tile 对象（ScriptableObject）**，本身不含图像，靠 **GUID 引用**源图里对应的 sprite。

**版权连带后果（重要）**：因为整个 `pixel-art-top-down/` 文件夹被 `.gitignore` 排除，那个**带切割信息的 meta 也一起没入库**（实测 `git ls-files` 该目录 = 0）。所以：

| 文件 | 存什么 | 入库 |
|---|---|---|
| `TX Tileset Grass.png` | 像素图（Cainos 版权） | ❌ ignore |
| `TX Tileset Grass.png.meta` | 256 个切割矩形坐标 | ❌ ignore（同文件夹） |
| `GrassTiles/*.asset` ×256 | Tile 对象，GUID 指向上面的 sprite | ✅ 入库 |

队友 clone 后会拿到 256 个 `.asset`，但拿不到源图和切割 meta → 这些 Tile 会显示**丢失引用**，需各自下载 Cainos 包、放到相同路径、并重新按 16×16 切（或单独获取切好的 `.png.meta` 覆盖）。这是「不入库版权素材」的固有代价，对个人开发无影响。

> 文件夹的兄弟 meta 也要排除：`pixel-art-top-down.meta` 不在文件夹内（是文件夹的兄弟文件），需在 `.gitignore` 里单独加一行，否则会出现「有 meta 没文件夹」的 Unity 警告。
