## 二、方向偏移问题（Direction Bias）

### 2.1 现象描述

PixelLab Character 模式生成 8 方向 rotation（S / SE / E / NE / N / NW / W / SW）后，文件名是它"声称"的方向。但实际打开图看：

- 文件名 `south.png` 里的人物，**身体并不是真正面**——通常是 **3/4 偏一点 west**（身体微微转向画面左）
- 文件名 `south-east.png` 里的人物，**反而是真正的正面 S**（双肩水平、披风对称、胸前装饰居中）

整圈方向**整体往逆时针偏了一格**。

### 2.2 我（agent）的最初误判

第一次审查时，我用**脸的朝向**来判断方向：脸看向哪边就当人物朝哪边。结论是"south 是完美正面"。

用户**当场指出**："为什么我始终觉得 south 偏一点 west，south-east 更正呢？"

我让用户指认两张图，他选了两张让我**脱离文件名**重新判断。我看完之后说"第二张更正"——然后用户揭示那两张正是 south.png 和 south-east.png。

**我看反了**。

### 2.3 校准：必须用身体口径，不用脸口径

人在 3D 转身时**头部和躯干会有 yaw/twist 错位**——脸可以独立转向镜头，但身体的开度才是"角色面对哪个方向"的真实信号。Unity 里的方向逻辑是按**身体朝向**判断的（决定攻击范围、走向、面板朝向），不是按脸。

正确的方向判断口径（按优先级）：
1. **双肩连线** —— 完全水平 = 正面（S 或 N）；倾斜越大 = 越偏 3/4
2. **胸前装饰 / 腰带扣 / tabard 中心** —— 居中 = 正面；偏左/右 = 偏 east/west
3. **披风左右对称性** —— 对称 = 正面；只一侧露出 = 身体转向另一侧
4. **双脚朝向** —— 双脚平行对称 = 正面
5. **脸朝哪只是补充信号**，不能当主判据

### 2.4 横向验证：是个例还是系统性？

为了确认是 PixelLab 的系统性 bias 还是单次抽卡偶发，我们今天审了 6 个不同风格的全 8 方向 batch（共 48 张图），逐张按身体口径核对：

| Batch | 文件名 south 实际朝向 | 文件名 south-east 实际朝向 |
|---|---|---|
| Disgaea/Phantom Brave | 偏 SW（身体微转左） | 真 S（正面） |
| Final Fantasy Tactics Advance | 偏 SW | 真 S |
| Langrisser II | 偏 SW | 真 S |
| Octopath Traveler HD-2D | 偏 SW | 真 S |
| Sea of Stars | 偏 SW | 真 S |
| Tactics Ogre | 偏 SW | 真 S |

**6/6 一致**。这是 PixelLab Character rotation 模板的**系统性方向 bias**：整圈方向标签**逆时针偏一格**。

### 2.5 实践结论：Unity 映射要整体逆时针挪一格

**文件名 → Unity 实际方向映射表**：

| 玩家输入 | 真实方向 | 应该用哪个文件 |
|---|---|---|
| ↓ | S | `south-east.png` |
| ↘ | SE | `east.png` |
| → | E | `north-east.png` |
| ↗ | NE | `north.png` |
| ↑ | N | `north-west.png` |
| ↖ | NW | `west.png` |
| ← | W | `south-west.png` |
| ↙ | SW | `south.png` |

**所以"PixelLab 文件名不可信"是一条硬结论**。后续每个 batch 的导入都要做这一格旋转。

### 2.6 工程含义

- v2 prompt 文档要加一节：**"PixelLab 方向 bias 与 Unity 映射规则"**
- 写一个导入脚本（或文档约定），把 PixelLab 输出的 8 个文件**按上表自动重命名**，再喂给 Unity sprite atlas
- 任何"评审是否生成正确"的工作，都必须先脱离文件名、按身体读

## 三、Hood + 长发的物理冲突问题

### 3.1 苏瑶的视觉设定

- 长银发，及背或更长
- 蓝色 hood（兜帽）+ 蓝色 cape（披风）
- 战斗时 hood 通常**放下**（露出银发），但有些状态（潜行/魔法吟唱）hood **戴起**

也就是同一个角色需要两个变体：**hood up** 和 **hood down**。

### 3.2 之前一直没搞定的 3 个问题

#### 问题 A：hood up 时长发完全不见

`hood up` 的 prompt 写完，模型直接把头当成秃头处理——hood 下面什么都没有。从背面看 hood 后面应该垂下一片长银发，但模型输出的是"hood + 一个空空的脖子"。

#### 问题 B：直接说 "hood down" 会把 cape 一起脱掉

prompt 加 `hood lowered down`、`hood off`、`no hood` 之类，**模型把整件斗篷都脱了**——人物变成只穿 robe 的状态，cape 也消失。

**根因**：模型把 "hooded cloak" 当**单一服装**处理。它的世界里 hood 和 cape 是同一件衣服的两部分，没有"hood 翻下来但 cape 还在身上"这种概念。

#### 问题 C："只生成头发" 的 prompt 出来的是胸像

为了绕过问题 B，我们试过只在 prompt 里写**头部相关**信息（"long silver hair flowing past shoulders, head fully visible, no hood"），结果模型生成出来的就是**只有头和肩膀的胸像**——身体直接没了，因为我们的 prompt 里没有提身体。

### 3.3 三种解决方案（今天全部试通）

#### 方案 1：State 模式（character-state）

PixelLab 提供 character-state API：先生成一个 base character（hood up 版本），然后用这张 base 当 reference，再生成一个"衍生状态"（hood down）。

**为什么能解决问题**：state 模式是 reference-conditioned 的——模型不是从 0 生成，而是**在已有图基础上做编辑**，identity（脸、发色、衣服整体）会被自动保留。

**实测**（`art/tmp/suyao_hood_nohood/hood_lowered_down_on/`，canvas 252）：
- prompt: `hood lowered down onto upper back behind shoulders, hood no longer covering head, head and forehead fully visible, long silver hair flowing from full scalp past shoulders to chest level, hair bangs over forehead, eyes nose mouth clearly visible`
- 结果：**8/8 全部成功**——hood 正确翻下来挂到肩后，长银发从头顶完整流下，cape 仍在，identity（脸型、发色、衣服）和 base 完全一致

**优点**：
- 完美保留 identity
- 解决 hood-cape 解耦
- prompt 简短（只描述差异）

**缺点**：
- 每次衍生消耗 20-40 token（贵）
- 必须先有 base 才能衍生（不能独立生成）
- 不能从 hood-up state 一步跨到完全不同的 outfit（差太大）

#### 方案 2：Mantle / Cape 词汇替换（避开 "hood" 这个词）

既然 "hood" 这个词在模型里和 "cloak" 强绑定，那就**根本不用这个词**。

把整套服装重新描述：
- "hood" → 删除
- "cloak" → 拆成 `attached blue mantle` (披在肩部到胸的短披) + `cape` (从肩到膝的长披)
- 加 z-order 显式约束：`silver hair drawn IN FRONT OF and OVER the shoulder mantle and cape`

**实测**（`art/tmp/direct_hood_down_v2/Fire_Emblem_GBA_style_chibi/`，canvas 244）：
- 8/8 全部成功——头完全露出，长银发从头顶流下盖在 mantle 之前，cape 依然完整
- 这是**今天最干净的成功 batch**——不需要 state、不需要 base、独立生成就成

**优点**：
- 一次生成、不需要 base
- 比 state 模式便宜
- silhouette 干净，hood 元素彻底消失

**缺点**：
- 要同时也做一个 hood-up 版本时，需要**完全重新写 prompt**
- 失去了"同一角色不同状态"的关联性（两个 prompt 完全独立）

#### 方案 3：Cowl 替换（用 cowl 当 hood 的替代词）

用 `cowl`（高领围巾、大翻领）替代 hood 的位置。cowl 在概念上是"绕颈不上头"的，能从根本上避免"覆盖头部"。

**实测**（`art/tmp/direct_hood_down_v2/Fire_Emblem_GBA_style_chibi_cowl/`，canvas 240）：
- 8/8 全部成功
- 但**cowl 被模型重解读了**——原意是脖子上的高领围巾，模型给的是**蓝色发饰带**（戴在头上、盘在头发上方那种）
- 视觉效果**很好看**——意外地像一个高级的发饰

**优点**：
- 8/8 成功
- 视觉效果意外地好（高级感）

**缺点**：
- cowl 意图被改写（从围巾变成发饰）
- 不再适合"hood up"语义——cowl 永远不会"盖住整个头"

### 3.4 用户给的关键洞察

讨论 hood 问题时，用户提了一句：

> "或许说后面的头发要在 HOOD 之上"

这句话直接命中根因——之前的失败不是"hood 没下来"而是"**z-order 没显式说**"。模型默认 hood 是最外层（覆盖所有头发），所以即便我们说 "long hair flowing"，hair 也被 hood 盖在底下。

**修正手段**：在 prompt 里写显式的 z-order 规则：
```
silver hair drawn IN FRONT OF and OVER the shoulder mantle and cape
```
（"hair drawn IN FRONT OF mantle/cape" 是一个**绘画指令而不是物理描述**——直接告诉模型图层顺序）

加了这条之后方案 2 就成了。

### 3.5 三方案对比表

| 维度 | State 模式 | Mantle/Cape 替换 | Cowl 替换 |
|---|---|---|---|
| 一次成功率 | 8/8 | 8/8 | 8/8 |
| Token 成本 | 高（20-40/state）| 标准 | 标准 |
| 是否需要 base | 是 | 否 | 否 |
| Identity 保留 | 完美 | 不适用（独立生成） | 不适用 |
| 与 hood-up 互通 | 天然互通 | 需重写 prompt | 需重写 |
| 推荐场景 | "同一角色多状态"（hood up/down 都要） | "只要 hood-down 一种状态" | 备选风格款 |

### 3.6 当前选择

**苏瑶日常形态（hood-down 主形态）**：方案 2（Mantle/Cape 替换）—— 最干净最便宜
**苏瑶吟唱/隐匿形态（hood-up）**：用方案 1（state 模式），从主形态衍生

## 四、Canvas 尺寸的副决定

之前一直在 248 上跑，今天一并搞清了 canvas 尺寸对**方向准确度**的影响。

| Canvas | 现象 |
|---|---|
| ≤156 | 方向标签准确（south 真是 south）；但分辨率太低，细节糊 |
| 156-244 | 过渡带，方向有时偏有时不偏 |
| **244** | **细节够、方向 bias 已稳定（整圈逆时针 1 格）、可以靠映射表纠正** |
| 248-252 | 同 244，差别不大 |
| >252 | bias 加剧、有时还会出"3/4 视角 + 22-45° 顺时针 skew"等额外形变 |

**v2 主 canvas 选 244**（128px 角色 sprite 用 PPU=64 时刚好对齐）。当 canvas 必须更大（hood-down state 等）时用 248-252，方向校正方法不变。

### 5.1 候选风格

我们用**完全相同的角色 prompt**（mantle/cape 版苏瑶），只换 style descriptor，跑了 6 个风格 8 方向 batch：

- **G = v2 mantle**（Fire Emblem GBA chibi + mantle/cape 命名版）：`direct_hood_down_v2/Fire_Emblem_GBA_style_chibi/`

### 5.2 横向评分

评审约束："只看风格，不管 hood 状态"（hood 已经另解，本节不重复）。
评审口径：身体读法。

| 风格 | 头身 | 配色层次 | 战斗感 | 记忆点 | 与"chibi 苏瑶"贴合 | 综合 |
|---|---|---|---|---|---|---|
| **G. v2 mantle (FE GBA chibi)** | 3.0 | 中（白蓝+小金） | 中 | 金扣纹章 | ⭐⭐⭐⭐⭐ | **平衡之选** |
| **F. Tactics Ogre** | 3.5-4 | **高（蓝+红+金 三层）** | **强** | 红 tabard + 金肩甲 | ⭐⭐⭐⭐ | **新王者**（识别度最高）|
| D. Octopath HD-2D | 3.0 | 高（蓝+金） | 中 | **星空披风**（极强）| ⭐⭐⭐⭐ | 视觉冲击王，主题漂移成"星空法师" |

| A. Disgaea | 3.0 | 中 | 弱 | 无 | ⭐⭐ | 偏低幼，淘汰 |

### 5.3 关键观察

#### 观察 1：模型对"风格 tag"的解读偏向"画风+主题"打包

我们以为只换"画风"，但 prompt 里给的"Octopath" tag 把**主题也带过来了**——模型生成出"星空披风"，因为它训练数据里 Octopath = HD-2D + 奇幻 + 星空 + 神官。FFTA 同理，把**整个色调拉成 FFTA 那种水蓝**。

**含义**：将来想"只换画风不换主题"，需要在 prompt 里**显式覆盖**主题元素（明确写"NO star pattern, NO blue-only palette"等）。

#### 观察 2：写实风（Langrisser / Tactics Ogre）会自动把头身比往上拉

只要 style descriptor 一带"严肃战棋"色彩（Langrisser、Tactics Ogre、FFT），模型会**自动忽略 prompt 里的 "3-head chibi"约束**，把比例拉到 4-5 头。

**Tactics Ogre 是个例外**——它停在 3.5-4，仍在 chibi 阵营。这可能是因为 "chibi" 在它的训练分布里有更多正样本。

**含义**："chibi" 这个词不是硬约束。要稳住 3 头身需要靠**风格本身的训练分布**（FE GBA、Octopath、Disgaea 这种本就是 chibi 的）来锁。

#### 观察 3：v2 mantle 是唯一一个"全可控"的风格

它头身比对、配色对、与 prompt 描述的细节（金扣纹章、白蓝色、长银发）100% 兑现。其他风格多多少少都有"模型自由发挥"的部分。

#### 观察 4：Tactics Ogre 是潜力黑马

完全没预期它能保住 chibi 比例，结果它做到了，而且配色层次（蓝+红+金）和战斗感（金属肩甲）**比 v2 mantle 还强**。

### 5.4 决策状态

**主形态候选缩到 2 个**：
1. **G. v2 mantle** —— 最 chibi 最可爱，配色简洁，与"小法师"定位贴
2. **F. Tactics Ogre** —— 配色最丰富，识别度最高，与"主角法师"定位贴

**决策悬而未决**——取决于苏瑶在游戏里的角色定位：
- 萌系 chibi 小法师 → G
- chibi 化的女主角法师 → F

剩余 5 个候选的处置：
- **D. Octopath**：保留作为视觉资产，可能用作苏瑶**觉醒/终极形态**
- **B. FFTA**：作为"治愈系/支援"备选风格资料
- **E / A**：弃用
- **C. Langrisser II**：弃用为主形态，可作为"成年/觉醒形态"参考资料

1. **R1 hood-down 的脱身体问题**——我以为成功，用户指出图里只有头没有身体

- **批量审图时容易掉回脸口径**——必须强制每张图先看肩、再看 cape 对称、再看脚
- **不要急着下宏观结论**——尤其在已经有过一次错的情况下
- **被用户质疑时，先重看图，不要嘴硬**——用户已经累计教过 3 次，trust capital 是底线，不容再耗

将来的方向判断 SOP：
1. 完全不看文件名
2. 看双肩连线水平度
3. 看胸前装饰是否居中
4. 看披风左右对称性
5. 看双脚朝向
6. 脸朝向只作为**第 5 优先级补充**信号
7. 给方向打标签时，**写出判断依据**（"双肩水平 + tabard 居中 → 真 S"），不要只写结论
