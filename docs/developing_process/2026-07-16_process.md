## 0. 这个 session 最重要的沉淀（一句话）

**同一个系列的人物，「比例得一样、方向得一样」才像一套。** 而这两条里：
- **「方向一样」= 每个角色 rotation 出来后，人工按【视觉】敲定 6 槽 anchor**（AI 绝不预设，PixelLab 朝向每个角色随机）。
- **「比例一样」= 头 size 一样最重要，头身比其次**（`头大小 > 头身比`）。地图上 sprite 摆一样大，玩家第一眼扫的是**头**；头一致就像同一套，身高长短随意。

## 1. 把资产搬进 `pixellab/` folder（目录约定定型）

以前 PixelLab 的产出散在各处、命名用 pixellab 自己的 `south/east/...`（不稳定、每角色不同）。这个 session
把它**收拢成一套固定目录**，并确立「真值源 + 派生物」的分层：

```
pixellab/
├── character_status.yaml            ← 真值源（每角色 prompt / character_id / anchor / 每动画 prompt+status）
└── characters/<id>/
    ├── download/     ← 原始 zip + 解压件（GITIGNORED，可随时由 character_id 重下）
    ├── review/       ← 逐帧对比 PNG / filmstrip（人工审用）
    ├── rotations/    ← 8 方向原图（pixellab 命名）+ _contact.png
    └── animations/<action>/<S|N|E|W|SE|SW>/frame_00N.png   ← 【我们的方向】命名（按 anchor 映射）
```

关键决定：
- **`download/` 一律 gitignore** —— 它能随时由 `character_id` 重新拉 zip 解压，不进版本库。
- **`animations/` 的方向名由 anchor 决定**，脚本读 yaml、**不写死**。pixellab 帧 `south.png` 会被 copy+改名成
  「我们的方向」`S/N/E/...`，Unity 里那个文件夹**就是**我们的方向，来自哪张 pixellab 帧不关心。
- 配套的可复用小脚本（本 session 沉淀）：`_refresh_download.py`（按 yaml char_id 重拉+解压）、
  `_organize_web.py`（网页下载按 anchor+镜像整理，支持碎片 merge）、`_filmstrip.py`（出一张逐行审阅图）、
  `_scan_anims.py`、`measure_head.py`。

## 2. 方向怎么定（两套场景 + 6 方向 + 只生成 4 个源）

### 2.1 两套场景 → 6 个游戏方向
游戏有两种镜头，各要不同方向：

| 场景 | idle | walk | attack | knockback / knockout |
|---|---|---|---|---|
| **地图**（top-down） | S | S N E W | — | — |
| **战斗**（45° 侧视，FE 风） | SE / SW | E W | SE / SW | SE / SW |

合起来游戏要 6 个方向：`S / N / E / W / SE / SW`。

### 2.2 只【生成】4 个源，其余镜像
- **生成源** = `S`(视觉正面) / `N`(视觉背面) / `E`(视觉侧向, walk 用) / `SE`(视觉 45°, 战斗用)。
- **镜像** = `W = flipX(E)`、`SW = flipX(SE)`；S/N 无镜像。
- **E 常复用 SE**：很多角色没有干净的近侧脸（纯 90° profile 会崩），就让 `E` 直接指向 45° 那张（E 和 SE 同一帧），
  **不额外生成**。本 session 里绝大多数角色都是 E 复用 SE；EmpireCaptain 是唯一 E≠SE（有独立侧向）。
- **哪一侧是真图 / 哪一侧镜像由用户拍板**（别默认 W 一定是 E 的镜像）—— EmpireHeavyCavalier 就因为 pixellab 的
  45° 帧朝左，反过来 W/SW 才是镜像。

### 2.3 怎么定方向 —— 黄金铁律：人工按【视觉】锚定
**PixelLab 的 8 方向朝向不稳定、每个角色随机，没有固定偏移公式**（历史上「逆时针偏一格」是错的，已废弃）。所以：

1. **只跑 rotation** → 出 8 张图 + 一张 `_contact.png` 对照表。
2. **停下来，用户看图**，按图里**身体/武器的真实朝向**（不看脸、**不认 pixellab 文件名**）指定：
   每个【我们的方向】= 哪张 pixellab 图。一张名叫 `south` 的图视觉上可能是 SE、`east` 可能才是干净 45°。
3. anchor（6 槽）写进 yaml。**anchor 没定，绝不允许跑 animation。**
4. 一个角色的 anchor **不能套用**给别的角色。

> 本 session 反复踩到的坑：EmpireHeavyCavalier 的 `north` 帧把骑士画成「半个人 / 握剑手糊掉」，
> 只有 pixellab `south` 帧干净 → 于是 E 和 SE **都锚 `south`**，W/SW 用我们自己的 flipX 补
> （pixellab 自己镜像不了 `south`）。这就是「按视觉、不认文件名」的活教材。

### 2.4 哪些动作做哪些方向（`dirs`）
写进每个动画的 `dirs`（游戏方向，生成时按 anchor 反查 pixellab 源）：

| 动作 | dirs（生成源） | 落盘后（含镜像） |
|---|---|---|
| idle | `[S, SE]` | S / SE / SW |
| walk | `[S, N, E]` | S / N / E / W |
| attack | `[SE]` | SE / SW |
| knockback | `[SE]` | SE / SW |
| knockout | `[SE]` | SE / SW |

村民（NPC）只做 `idle + walk`（无战斗）。角色可加专属动作（如 captain 的 `defense`）。

## 3. 怎么定 sprite 的 size —— 头 size 最重要

QA 有两条硬指标，过不了就重出：

### 3.1 头长 45–48px（这是「一套感」的命门）
- **`头大小 > 头身比`**：地图上所有 sprite 摆成一样大，玩家第一眼扫到的是**头**。**头一致 → 像同一套**；
  身高长短随意（女生矮、男主腿长都 OK，身体高度弹性放开）。
- 友方主力**必须**落在发顶→下巴 **45–48px**；敌兵可略放宽。
- ⚠️ **兜帽 / 头盔 / 发髻会把「发顶」虚高 3–4px**，量的时候心里扣掉。
- 工具：`measure_head.py` 放大打网格 + 目测发顶/下巴线做**粗筛**（±2–3px）；**最终精确值用 Aseprite 手量**
  （`rotations/*.png` 拖进去、选框数发顶→下巴行数）。

### 3.2 正 S / 正 N / 侧面 三向 rotation 无问题
正对 / 正背 / 干净侧向这三向是地图与战斗的主力帧，武器与身体朝向要对、silhouette 不崩（其余方向靠镜像/复用）。

### 3.3 其它 size 相关约定
- `image_size` 保持 **128**；rotation 用 **3-头身 super-deformed chibi**。
- **不用 "GBA" 老式像素词**（人物粗糙 + 强绑一撮翎羽），也**别加 "modern/high-res pixel"**（乱长金犄角）；
  尾巴走默认 `clean pixel art, vibrant colors, sharp silhouette`。

## 4. 骑士 / 骑兵怎么考虑（骑马单位的特殊处理）

骑兵是这个 session 最麻烦的一类，踩了不少坑，沉淀出几条：

- **长枪（lance）必须写「large / long / oversized」**，否则 PixelLab 画得又细又小。
- **一把枪只提一次**：早期 EmpireCavalier prompt 里重复描述长枪，结果画出**第二把半截枪 / 双枪**。收成
  「a single large long cavalry lance upright in the right hand」一处描述后才稳。
- **候选筛选很关键**：EmpireCavalier 一批 8 版里，#3–#7 多是**半截枪 / 双枪**，另一版 N 方位**长枪穿模**、S 方位枪歪，
  只有一版 S/N/SE 全对且不穿模才能用。骑兵的鞍/枪/马腿容易崩，**必须逐版看、宁可重出**。
- **马的 walk 是「马的步态」不是人的**：walk prompt 要写「the horse's legs never all planted / forward stride +
  rider gentle bob / no galloping / no rearing / no sliding hooves」。
- **骑兵的 knockout = 人从马上摔下**（马踉跄、骑手被甩落砸地），不是人自己倒。
- **带甲马的「骑士长」留作未来升级版**：一阶小兵骑兵用较轻的甲、无马铠即可（tier-1 mook）。

## 5. 动画 prompt 工程的实测结论（本 session 定稿）

- **F1 绝不写 idle**：PixelLab 一定把 rotation 帧塞成 **frame_000（idle 静止起始）**，`frame_count=8` 落盘其实
  **9 帧**。所以 8 帧描述要**直接从动作开始**，否则开头僵两帧。
- **knockback（后仰）是最难的**：PixelLab「被打」先验是**向前缩/前扑**，凡用 `smashed / violently / falls /
  prone / to the ground` 这类**灾难级+倒地词**一律被掰成向前栽、"backward" 被无视。**诀窍 = 措辞克制 +
  结尾回正站立**（`recoils, slightly bent backward` + `full return, upright`，不倒地）。
- **knockout（倒地）**：允许倒地，PixelLab 会**向前**倒，做阵亡可接受。
- **sprite 上绝不烘焙 VFX**：施法/攻击**不画**发光/闪光/法术光/粒子，用重否定词（`NO glow, NO light,
  NO sparkles ...`）、法器写 `plain / unlit`；VFX 后期独立叠加。
- **两条新约定（本 session 后期，07-19 定，仅对之后写的新 prompt 生效）**：
  1. **不再加 `Clean pixel art, sharp silhouette.` 尾巴** —— PixelLab 自己会控质量，加了没用还占 1000 字符额度、污染。
  2. **knockout 简化 = 「倒地、闭眼」就完了** —— **去掉武器散落**（`weapon fallen near him` 效果不好），
     只留「摔倒 → 趴地 → 双眼闭上不起来」。

## 6. 两条生成路径（脚本 A / 网页 B）

- **路径 A（脚本）** `pixellab_gen_animations.py`：一动作一次 `create_animation_v3`（该动作所有源方向一起提交），
  从每个 job 的 `last_response.images`（9 张 rgba_bytes）**直接解帧**落盘，再按 anchor 的 `mirror:X` 做镜像。
  ⚠️ 脚本路径的 **zip 会碎片化**（同名多 call → `walk`/`walk-<hash>`），所以脚本路径**不从 zip 组织**，job.images 才是真源。
  ⚠️ 该脚本每个动作开头会 `rmtree(animations/<action>)`，所以 `--only-dirs` 补单方向会**误删同动作其它方向**——
  补单方向要用一次性单方向小脚本（不 wipe）。
- **路径 B（网页）**：用户在 PixelLab web 上把各方向合进一个动画组 → zip 干净 → `_refresh_download.py` 重拉 →
  `_organize_web.py` 按 anchor copy+改名+镜像。**网页 walk 可能拆成两个文件夹**（如 `walk`[north-west] +
  `walk-<hash>`[east,south]）→ `_organize_web.py` 已加 **merge**：一个 action 收集所有同名文件夹、跨文件夹找源方向。
- **API 不能改已有动画组**（请求体只有 `animation_name`+`directions`，无 `replace_existing`）→ 同名重发 = 新建碎片。
  网页 UI 能「组内加方向/重生成/镜像」，API 不行。

> 教训（本 session 真实事故）：一个「finalize」临时脚本先 `rmtree(animations)` 再 `copytree(旧base)`，第二步失败
> → animations 被删。**所幸 API 生成的动画永远能从角色 zip 重新拉回来** → 重新 organize 即恢复。
> **凡是要动 animations/ 的临时脚本，先想清楚删了能不能恢复。**
