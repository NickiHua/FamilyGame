## 一、瓦片边缘方案选型:从四方案到 dual-grid

和用户把「地形交界怎么过渡」从头捋了一遍,核心是看懂 **Wang tile 家族**:

- **Wang 是个家族**(数学家王浩),按「匹配点在哪」分三支:**边匹配(edge)** / **角匹配(corner)** / **边+角匹配(2-edge 2-corner)**。
- **47-blob** = 2-edge-2-corner 的**47 张可用子集**;运行时其实**不是边色榫卯**,而是「**每个地形查自己 8 邻居的 同我/非我 bitmask**」(self-vs-other 二值),靠**分层(优先级)** 把高地形 overlay 压到低地形上。**每地形一套 47**,费在「47 张严格对齐」。
- **dual-grid** = 角匹配。**显示网格偏移半格**,每个显示 tile 查它压住的 **4 个角**(2^4=16)→ 从 **16 张**里选。逻辑就是个 **16 项查表**,完美契合「Python 按 JSON 渲染」。
- **硬边(现状)** = 不属 Wang,纯铺,直角缝。

### 关键认知(写给未来)
- **blob 的过渡靠「overlay 遍每格都查邻居」**:道路格自己查到「左右有草」→ 画一张「左右探草、中间透明」的草 overlay 盖在自己道路上,**探进来的草边就是过渡**。这也正是要凑齐 47 的原因。
- **分层只让「高地形溢到低地形」**(靠一张优先级表打破对称),否则两边互溢糊成泥。
- **dual/blob 都用分层**:总数 = `每地形张数 × (N-1)`。dual **16 vs blob 47**,且 16 旋转去重后**独立形状只 ~6**(blob ~15),几何还更简单(只编码 4 角,不含内凹)。→ **dual 同结构下稳赢,选 dual。**
- **互补残片**(用户最初设想:两张透明残片拼满一格)= overlay 的对称理想版;实战偷懒成「**低地形铺满底 + 高地形出透明渐隐残片**」,残片对任何底都能拼,**不必逐对设计**。

## 二、dual-grid 分层渲染器 `render_seg_dual.py` + 两个真 bug

写了 `scripts/tiles/demo_transitions.py`(hard / dual / 47-blob 三方案对比图 + **dual 16 张全集 sheet**,四角点绿/蓝点标注「哪个角是草/水」)让用户直观看懂,再写 `render_seg_dual.py` 把现有 seg 地图用 **dual-grid 分层**重渲(地面圆角过渡,树/房/桥作 object 盖上)。

### ⚠️ bug 1:桥两端「水跑岸上」
桥的棕色格在 seg 里**横跨整条**,两端其实压在草岸上。我一开始把**所有** bridge 格当 water → dual 从桥头把水**鼓成楔形**渗进草地。
**修**:bridge 格**只有四邻有真 water 才算 water**,在岸上的当 grass。

### ⚠️ bug 2(真凶):地形缝「凭空冒水」
画布**整张先铺成水**当底再盖地形 → **路/石/草三种高地形交汇的缝**里谁的 field 都没到 0.5 → **露出底下的水**,凭空蓝水。
**修**:底改成**每格先按自己地形硬铺**,再 dual 圆角覆盖。**没被覆盖处露自己格的地形,不漏水。** (教训:overlay 合成**别用单一兜底色当全局底**,要用 per-cell 硬底。)

### enrichment:宏观明暗 + decal(逐格抖动翻车)
为追梦战4 那种「大片丰富纹理」,加两层:**①宏观明暗**(全图低频噪声乘,大片明暗起伏)**②非对齐 decal 撒点**(石/草/浪花)。
- **逐格亮度抖动 = 方块棋盘**(每整格一个亮度,边界突变)→ **弃**。要大范围变化必须用**平滑跨格的宏观层**,不是 per-cell。

## 三、FE/梦战 参考:悬崖是另一套,不进 dual-grid

- **视角 = 3/4(斜俯视 / oblique top-down)**:主俯视但立面(山壁/屋墙)露一点正面。
- **悬崖 ≠ 平面过渡**:它是「**高度落差 + 朝镜头的正面墙**」,强方向、第三材质。**塞进 dual 的角查表必拧。** 正解:**地表走 dual-grid;有高度的立面走单独的「方向性墙 autotile」(直墙段重复 + 内外拐角),只在朝镜头的落差边渲染;树/石/房/桥走正面 object sprite。**
- 梦战平面海岸(无悬崖)正是 dual-grid 主场,所以比火纹「容易一档」。

## 五、像素美术认知(和用户掰扯透的几条)

- **像素感 = 硬边(无抗锯齿)+ 原生分辨率 + Point 显示 + 整数缩放**。四者缺一糊。
- **「像素感来自相邻像素值的硬跳变」**——用户的模型对:`0 0 255 0 0`(脆) vs `0 128 255 128 0`(那俩 128 是 AA 过渡像素,糊)。
- **有限调色板不是必须,是风格 + 量化时的省事杠杆**。真彩像素画成立(现代 HD 像素)。**杀死像素感的是「羽化/AA」,不是「多色」。**
- **同 64×64 缩到同尺寸,像素感可不同**;**缩到越少像素 → 块越大 → 像素感越强**(32 比 64 更 chunky)。
- **AI 直接出真像素 = 走不通**(GPT 天生满图 AA + 不守尺寸)。要真像素只能 **AI 出形 → 后处理(下采样+量化+去 AA 掰硬)**,或程序化/手绘。
- **Aseprite vs Python**:手工精修/修缝/调色板 → Aseprite 强(人眼在环);批量/可复用下采样量化无缝 → Python 强。**Aseprite 能量化(Indexed/Reduce palette),但只动颜色不重画边形、不管无缝**;「resize 就保证像素感」是夸大,缩出来是糊脏低分辨率,真脆要靠量化+手修。

## 六、tile 出图实验矩阵 → 定标准 ⭐(本 session 最大产出)

跑了一整套对照(都 opaque、1024 原图、再下采样):

| 批次 | prompt 尺寸词 | quality | 落点 |
|---|---|---|---|
| 64_low | 64x64 | low | grass/road/water(首批 3 种) |
| 32_low | chunky 32x32(**清爽强调**) | low | 5 种 |
| 32_detail | chunky 32x32(**自然细节**) | low | 5 种 |
| 32_med / 64_med | 32 / 64 | medium | 5 种 |
| 32_high(全) / 64_high(grass) | 32 / 64 | high | |

### 结论(全部已肉眼亲验)
1. **prompt 的「32x32」是「画得更 chunky」的杠杆,≠ 最终 tile 尺寸**。**「32-prompt 出图 → 下采样到 64」= 甜点**:chunky 像素味 + 64 的细节容量(配 128 sprite 正好 2:1)。这是**关键解耦**。
2. **clean 强调会翻车**:32_low 草**太平太死**、路**全成竖木纹**、桥**白边框**。**适度自然细节(detail)反而更稳**——过度抑制让 GPT 出怪东西。
3. **颗粒/整洁度:high > medium > low**(medium 已很好,high 边际更脆)。
4. **32-prompt 比 64-prompt 更像素**(64 更细偏 HD)。做像素 SRPG **32 胜**。
5. **finals 标准 = 32-prompt + HIGH + 下采样到 64**。主力 base 用 high(就几张),次要/变体 medium 够。

### 出图工艺约定
- **`docs/prompts/tile-prompts.md` 每次调 API 前先更新**(文件 = 即将调用的真实 prompt);本 session 在文件里留了批次 A–G 段。
- **GPT-Image 速率限制 5 张/min**:用**带自动重试(429→等 20s)的临时驱动**一把跑完一批,省得反复手动撞限速(`_gen_batch.py` / `_gen_high.py`,用完即删)。
- **成本**:low ≈ 1 分/张,medium 数倍,high ~ 1–2 角/张。base 量小,high 花得起。
