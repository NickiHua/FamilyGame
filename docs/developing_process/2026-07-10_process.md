# 2026-07-10 开发日志：Stage1 地图整图重绘实验（GPT Image 2.0 vs Seedream 5.0 Pro）

> 本篇记录一次集中地图视觉实验。目标是验证：在不改变 `stage1_map.json` 格子语义的前提下，能否用 AI 把当前 dual-grid / tilemap 地表重绘成更自然、更统一的日式手绘 SRPG 地图。
>
> 最终结论非常明确：**Seedream 5.0 Pro 的结构稳定性明显高于 GPT Image 2.0**。最佳路线是：**用 Unity 根据 JSON 渲染出的 dual-grid 视觉图作为第一参考图锁布局，再用 GPT / Gemini 生成的边缘 transition sheet 作为第二参考图控制水岸、草路、路泥等边缘语言，交给 Seedream Pro 重绘整张地图。**

---

## 一、核心目标

当前 Stage1 地图已经从纯手绘 Tilemap 进化到：

```text
Stage1 Tilemap -> Export Stage1 JSON -> MapGrid -> DualGridBuilder -> DualGrid 视觉层
```

但视觉上仍有问题：

- 原始 `Stage1` tilemap 太硬，边界明显是块状 tile。
- dual-grid 虽然能把边界变得有机一点，但整体仍然像规则 tileset 拼图。
- 水岸、草路、路泥过渡缺少一张手绘地图的整体统一感。
- 希望接近《尤特娜战记 / TearRing Saga》一类正俯视 SRPG 地表质感，但不要写年代词，以免 GPT 画得过老。

本次实验想验证一个生产管线：

```text
第一参考图：当前地图布局 / dual-grid 地形分布 / 可玩格子语义
第二参考图：边缘材质语言 / 柔和程度 / 水岸细节 / 草路泥过渡
输出：完整 1920x1920 地图地表图
```

要求：

- 保留河流、道路、泥地、草地的位置。
- 不添加树、房子、桥、石头堆、角色、UI、网格线、文字。
- AI 只负责视觉重绘，不负责重新设计地图。
- 最终仍能和 `stage1_map.json` 对齐。

## 二、准备参考图：raw tilemap、dual-grid、边缘 sheet

### 2.1 Dump 两张 1920 地图参考图

最初先从 Unity 里导出两张 1920x1920 地图图：

```text
art/maps/map_raw/stage1_tilemap_raw_1920.png
art/maps/map_raw/stage1_tilemap_dualgrid_1920.png
```

第一次误把整张 scene 视觉层都 bake 进去了，包括房子、树、石头、桥 sprite、overlay 等。后来修正为 **tile-only dump**：

- raw 图只包含 `Stage1` 的 `TilemapRenderer`。
- dual-grid 图只包含由 `DualGridBuilder` 生成的地表 tilemap 层。
- 不包含 houses / props / units / UI / scene objects。

这一步增加了工具：

```text
Assets/Scripts/Editor/MapImageDumper.cs
```

后续又补了一个专用入口：

```text
DumpStage1DualGridFromJsonImage
```

这个入口只读：

```text
Assets/Art/Maps/stage1_map.json
```

不会跑 `MapExporter.Export()`，因此不会把 Unity scene 中的旧 Tilemap 反向覆盖 JSON。这个点后来变得很关键。

### 2.2 生成边缘 transition sheet

为了给 AI 一个明确的「边缘风格参考」，让 GPT 先生成了几张 terrain transition sheet。主要文件：

```text
art/maps/map_raw/edge_transition.png
art/maps/map_raw/edge_transition_gemini.png
art/maps/map_raw/edge_transition_less_details.png
```

prompt 方向反复调整过：

- 不再写 90s / 2000s，避免 GPT 画得太老。
- 要求 **日式手绘 SRPG 地图风格**。
- 早期要求“不要柔和”，后来根据目标改成：**柔和渐变、低对比、不要太硬**。
- 最终做成 2x2 sheet，每格是一种两地形边缘：
  - 草水
  - 草路
  - 路泥
  - 路水

后续验证里，`edge_transition_less_details.png` 最适合作为 Seedream 的第二参考：细节密度更低，不容易把整图边缘画得太碎。

## 三、GPT Image 2.0 实验

### 3.1 API 能力确认

查官方文档后确认：

- GPT Image API 支持多参考图，不是只能一张。
- `gpt-image-2` 支持 `1920x1920` 这类自定义尺寸。
- `gpt-image-1.5` 仍是旧固定尺寸体系，不适合直接出 1920 方图。
- `gpt-image-2` 不支持透明背景，但本次地图是 opaque，无影响。

当前脚本本来已经支持多 `--ref`：

```text
scripts/gpt_transparent_image_generation.py
```

命令形态：

```powershell
.\.venv\Scripts\python.exe scripts\gpt_transparent_image_generation.py custom `
  --type map_redraw `
  --model gpt-image-2 `
  --quality medium `
  --background opaque `
  --size 1920x1920 `
  --ref art\maps\map_raw\stage1_tilemap_dualgrid_1920.png `
  --ref art\maps\map_raw\edge_transition.png `
  --prompt "..." `
  --out <output.png>
```

### 3.2 GPT Image prompt

新增英文 prompt：

```text
art_undecided/maps/stage1_gptimage_prompt_en_v1_full_multiref.txt
```

核心意思：

```text
第一张参考图只控制地图 layout / terrain placement。
第二张参考图只控制 terrain edge style / softness / color / blending language。
不要复制 2x2 sheet 的排版。
不要添加 trees / buildings / bridge / UI / text。
```

### 3.4 GPT Image 观察

GPT Image 2.0 的优点：

- 画面完成度高。
- 边缘柔化自然。
- `high` 质量下草地、水岸、道路细节更干净。
- 对“整张图统一风格”的理解不错。

问题：

- **不够守 layout。**
- 河岸、路边、泥地区块会被艺术化后膨胀或收缩。
- `high` 比 `medium` 更好看，但也更倾向重绘/美术化，边界漂移更明显。
- 用 raw tilemap 作为第一参考时，AI 会试图解释硬边，容易产生一圈边缘膨胀。
- 用 dual-grid 作为第一参考时 layout 更稳，但 GPT 仍会把边界重新理解成更圆、更流畅的自然形状。

GPT 候选里 layout 最准的是：

```text
dual + edge GPT medium
```

偏移最多的是：

```text
dual + less-details edge high
```

最终对 GPT Image 2.0 的定位：

```text
适合：生成 edge transition sheet / 美术风格参考 / 首轮审美探索。
不适合：要求严格保留 SRPG tile layout 的整图地图重绘。
```

## 四、layout 验证方法

为了判断“好看”和“守格子”不是同一件事，做了几种对比图。

### 4.1 半透明 overlay

最初做的是：

```text
生成图作为底图 + 原 tilemap 35% 透明叠上去
```

后来用户指出不清楚，于是反过来：

```text
原图不透明作为底图 + 新图 50% 透明叠上去
```

仍然不够直观，因为两张图颜色不同，透明混合容易误导。

### 4.2 红青边界图

做法：

```text
old map edges = red
new map edges = cyan
background = darkened new image
```

用途：看地形边界是否重合。红青分离越明显，说明边界越漂。

代表文件：

```text
art_undecided/maps/stage1_high_vs_dual_layout_edge_overlay_red_old_cyan_new.png
```

### 4.3 棋盘格 old/new

做法：

```text
240px 一格，old / new 交替显示
```

用途：看道路、河流、泥地区块在格子边界处是否断裂。

代表文件：

```text
art_undecided/maps/stage1_dual_edge_gpt_medium_layout_checker_old_new_240.png
art_undecided/maps/stage1_raw_edge_gpt_medium_layout_checker_old_new_240.png
art_undecided/maps/stage1_raw_edge_gemini_medium_layout_checker_old_new_240.png
```

### 4.4 闪烁 GIF

做法：

```text
old / new 两帧交替闪烁
```

用途：肉眼快速判断整体结构有没有漂。

代表文件：

```text
art_undecided/maps/stage1_dual_edge_gpt_medium_layout_blink_old_new_960.gif
art_undecided/maps/stage1_raw_edge_gpt_medium_layout_blink_old_new_960.gif
art_undecided/maps/stage1_raw_edge_gemini_medium_layout_blink_old_new_960.gif
```

最终验证经验：

```text
判断 layout：优先看棋盘格 + 闪烁 GIF。
判断边界漂移：看红青边界图。
不要只看普通 diff heatmap，因为它会把纹理变化也当成差异。
```

## 五、Seedream 5.0 Pro / Lite 实验

### 5.1 Seedream Pro 文档确认

查火山方舟文档后确认：

- Seedream 5.0 Pro 支持 **2-10 张参考图生成单图**。
- `1920x1920` 合法。
- Pro 不支持 `sequential_image_generation` 和 `stream`，不能传这两个参数。
- 建议参数：

```json
{
  "model": "doubao-seedream-5-0-pro-260628",
  "size": "1920x1920",
  "output_format": "png",
  "watermark": false
}
```

### 5.2 新增 Seedream 地图脚本

为了避免旧 portrait 脚本传 Lite-only 参数，新增：

```text
scripts/volcengine_seedream_map.py
```

特性：

- stdlib-only。
- 支持多 `--ref`。
- `image` 字段传单张或多张 data URL。
- 不传 `sequential_image_generation` / `stream`，兼容 Pro。
- 支持 `--response-format b64_json|url`。
- 默认模型：

```text
doubao-seedream-5-0-pro-260628
```

### 5.3 Seedream Pro prompt v6

新增：

```text
art_undecided/maps/stage1_seedream_pro_prompt_cn_v6_full_multiref.txt
```

核心约束：

```text
第一张参考图 = 严格主布局来源。
第二张参考图 = 只参考地形边缘材质语言、柔和程度、色彩关系和细节密度。
不要复制 sheet 排版。
不要改变河流、道路、泥地、草地的语义位置。
不要添加任何对象层。
```

相比 v5，v6 加强了：

- “第一张参考图是地图结构参考，不是风格上限”。
- “不要把第二张参考图画成 sheet”。
- “边缘自然化幅度控制在很窄范围内”。
- “不要新增桥、房子、树、石头堆”。

### 5.4 Seedream Pro 初版两张

输入：

```text
ref1: art/maps/map_raw/stage1_tilemap_dualgrid_1920.png
ref2: art/maps/map_raw/edge_transition.png
ref2: art/maps/map_raw/edge_transition_less_details.png
```

输出：

```text
art_undecided/maps/sdmap/stage1_seedream5pro_dual_edge_transition_1920_20260710_144849.png
art_undecided/maps/sdmap/stage1_seedream5pro_dual_edge_lessdetails_1920_20260710_144849.png
```

观察：

- `edge_transition` 版本边缘细节更多，水岸沙泥、小碎石、碎草更明显。
- `edge_less_details` 版本更干净，更安静，整体细节密度更可控。
- 两张 Pro 的 layout 都非常稳，明显比 GPT Image 2.0 稳。

### 5.5 Seedream Lite 对比

又跑了两张 Lite：

```text
art_undecided/maps/sdmap/stage1_seedream5lite_dual_edge_transition_1920_20260710_145411.png
art_undecided/maps/sdmap/stage1_seedream5lite_dual_edge_lessdetails_1920_20260710_145411.png
```

观察：

- Lite 第一张把中下部河道误解成大水域，layout 崩得比较明显。
- Lite 第二张守 layout 好一些，但边缘有明显亮边 / 白边感。
- Lite 不适合作为最终地图生产模型。

结论：

```text
Seedream 5.0 Pro > Seedream 5.0 Lite
```

### 5.6 后续 JSON 修改与 no-bridge 问题

过程中发现一个隐藏问题：`stage1_map.json` 里没有桥 `D`，但 dual-grid 图里出现了桥。

原因在：

```text
Assets/Scripts/View/MapGrid.cs
```

旧逻辑会自动推断桥：

```csharp
_rows = InferBridgeCells(rows.ToArray());
```

只要水格左右能连到道路，就把 `W` 改成 `D`。

这破坏了“JSON 是唯一真相”的原则。因此改为：

```csharp
public bool inferBridgeCells = false;
_rows = inferBridgeCells ? InferBridgeCells(rows.ToArray()) : rows.ToArray();
```

后续重新 dump 的 dual-grid 图不再凭空加桥。

代表输出：

```text
art/maps/map_raw/stage1_tilemap_dualgrid_no_bridge_1920_from_json_20260710_213635.png
art/maps/map_raw/stage1_tilemap_dualgrid_no_bridge_1920_from_json_20260710_225921.png
art/maps/map_raw/stage1_tilemap_dualgrid_no_bridge_1920_from_json_20260710_233223.png
```

并继续用无桥 dual-grid + less-details ref 跑 Pro：

```text
art_undecided/maps/sdmap/stage1_seedream5pro_newdual_nobridge_edge_lessdetails_1920_20260710_213641.png
art_undecided/maps/sdmap/stage1_seedream5pro_newdual_nobridge_edge_lessdetails_url_a_1920_20260710_232052.png
art_undecided/maps/sdmap/stage1_seedream5pro_newdual_nobridge_edge_lessdetails_url_b_1920_20260710_232052.png
art_undecided/maps/sdmap/stage1_seedream5pro_newdual_nobridge_edge_lessdetails_url_a_1920_20260710_233230.png
art_undecided/maps/sdmap/stage1_seedream5pro_newdual_nobridge_edge_lessdetails_url_b_1920_20260710_233230.png
```

### 5.7 response_format=url 的必要性

第一次用 `b64_json` 跑 1920 PNG 时遇到：

```text
http.client.IncompleteRead
```

原因：1920 PNG 的 base64 JSON 响应很大，HTTP 读响应时中途断流。

修正：改用：

```text
--response-format url
```

让 Ark 返回小 JSON + 临时下载 URL，再由脚本下载图片。

结论：

```text
Seedream 1920 PNG 生产一律用 response_format=url。
不要用 b64_json 承载大图返回。
```

## 六、最终结论

### 6.1 模型稳定性

本次最重要结论：

```text
Seedream 5.0 Pro 的 layout 稳定性 > GPT Image 2.0
```

更细分：

```text
Seedream 5.0 Pro：最适合整张地图重绘，强约束 layout。
GPT Image 2.0：画面完成度高，但整图 layout 容易美术化漂移。
Seedream 5.0 Lite：不够稳，可能误解水域和地图结构。
GPT Image / Gemini：适合做 edge transition sheet，不适合作为最终整图保结构模型。
```

### 6.2 最终推荐生产管线

```text
1. 在 Unity Stage1 Tilemap 里编辑地图。
2. Export Stage1 JSON。
3. 以 stage1_map.json 为唯一真相，生成 dual-grid 视觉参考图：
   art/maps/map_raw/stage1_tilemap_dualgrid_1920.png
4. 用 GPT Image / Gemini 生成 edge transition sheet：
   art/maps/map_raw/edge_transition_less_details.png
5. 用 Seedream 5.0 Pro 多参考图生图：
   ref1 = dual-grid 视觉图（锁 layout）
   ref2 = edge transition sheet（锁边缘风格）
6. response_format=url，output_format=png，size=1920x1920，watermark=false。
7. 用棋盘格 / GIF / 红青边界图验证 layout。
```

核心思想：

```text
JSON 决定地图语义。
DualGrid 视觉图把 JSON 转成 AI 能理解的地形布局参考。
Edge sheet 提供边缘美术语言。
Seedream Pro 负责最终整图重绘。
```

### 6.3 为什么不是直接喂 raw tilemap

raw tilemap 太硬，AI 会大量解释硬边，容易在边缘外扩或误判。

dual-grid 更适合作为第一参考：

- 地形语义仍来自 JSON。
- 边界已经比 raw 更自然。
- AI 更容易理解“这是最终地图形状”。
- Seedream Pro 能在这个基础上做手绘化，而不是重建地图。

### 6.4 为什么第二参考用 less-details sheet

`edge_transition_less_details.png` 的优势：

- 细节密度低，不会诱导整图过碎。
- 岸线柔和，不会出现过强白边。
- 更适合大地图整体重绘，而不是局部素材展示。

因此当前最推荐组合是：

```text
ref1: art/maps/map_raw/stage1_tilemap_dualgrid_1920.png
ref2: art/maps/map_raw/edge_transition_less_details.png
model: doubao-seedream-5-0-pro-260628
size: 1920x1920
output_format: png
response_format: url
watermark: false
prompt: art_undecided/maps/stage1_seedream_pro_prompt_cn_v6_full_multiref.txt
```

## 七、Unity / repo 工程结论

### 7.1 `SampleBattle.unity` 不应该承载生成态

中途发现 `SampleBattle.unity` 出现 40k 行 diff。原因不是地图 JSON，而是 scene 里保存了大量生成对象：

```text
DualGrid
ScatteredProps
BattleRunner
BattleCanvas
MapGrid
树 / 石头 / 生成 props
```

这些对象每次 build / scatter 都会改 fileID 和顺序，导致 Unity YAML 巨大 diff。

最后确认：

```text
如果地图语义已经导出到 stage1_map.json，SampleBattle.unity 的生成态 diff 可以 revert。
```

已执行过一次：

```powershell
git restore -- Assets/Scenes/SampleBattle.unity
```

以后原则：

```text
Stage1 Tilemap 是编辑画布。
stage1_map.json 是地图语义源文件。
DualGrid 是可再生视觉结果。
不要提交因为预览 / scatter / battle build 产生的 scene 大 diff。
```

### 7.2 DualGridBuilder 去掉 Seedream overlay 自动层

此前 `DualGridBuilder` 会自动加：

```text
stage1_seedream_v3_overlay.png
```

作为 `seedream_ground_overlay` sprite。这会干扰 tilemap 编辑和新的 AI 工作流。

已从：

```text
Assets/Scripts/Editor/DualGridBuilder.cs
```

移除自动添加 overlay 的逻辑。

### 7.3 MapImageDumper 的作用

`MapImageDumper.cs` 现在用于：

- dump raw tilemap 1920 图。
- dump dual-grid 1920 图。
- JSON-only dump dual-grid，不覆盖 JSON。

其中最重要的是 JSON-only 入口：

```text
DumpStage1DualGridFromJsonImage
```

以后只要 `stage1_map.json` 更新，就可以直接生成新的 AI 主参考图。
