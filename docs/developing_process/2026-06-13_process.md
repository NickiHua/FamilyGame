# 2026-06-13/14 开发日志：第一张 demo 战场地图——AI 出图管线打通（JSON → 控制图 → ControlNet → 成品）

## 一、当日目标

做《幻世纪》**第一张 demo 战斗地图**。目标画风：类似 langrisser 手游那种精致的 **HD-2D 地图**，20–35 格见方的战场。

核心要解决的不是"画得好不好看"，而是 **可控性**——河流、房子、桥、悬崖必须**精确摆在指定格子上**，并且**美术图和游戏逻辑网格同源**（画一次数据，既喂出图、又当 Unity 的 walkable 判定）。

最终交付：一张 2240×2240 的 painterly 战场背景图 `art/maps/demo_map.png`，外加一整套可复跑的"JSON → 控制图 → ControlNet 出图"管线脚本。

> **关键架构决策**：**废弃**之前 16px Cainos tilemap 拼地图的方案（仅用于别处），战场背景改成 **一张 AI 大图 + 独立逻辑网格** 的分层结构。

---

## 二、为什么是"JSON 优先"这条路（设计推演）

一开始有两种思路在打架：

1. **纯 prompt 生成**：直接让 Flux 出战场图。问题：**完全不可控**——河画哪、几座桥、房子多大，全靠运气，且对不上格子。
2. **JSON 优先（最终采用）**：先用数据定义每格是什么，再把数据翻译成"控制图"喂给模型，模型只负责画材质光影、不负责定布局。

GPT 当时也独立提出了 JSON 优先的思路，和我的"控制图与逻辑网格同源"是**同一件事**，只是它拆得更细：

```
第一步：JSON 定义每格地形（river/forest/houses/...）
第二步：structure render —— JSON 翻译成模型能读的控制图（PNG）
第三步：把控制图喂出图模型
```

**这条路最大的价值**：同一份 `demo_map.json` 同时喂两边——
- 给 Flux 出**美术图**
- 给 Unity 出 **walkable 逻辑层**

美术和逻辑**永不错位**。这是纯 prompt 路线永远做不到的。

## 三、写了哪两个脚本，各干什么（重点）

都在 `scripts/map/` 下，用项目自己的 venv（`c:/Repo/FamilyGame/.venv`，装了 pillow + numpy）跑。

### 3.1 `build_demo_map.py` —— 布局真值源（"画地图数据"）

**它干什么**：把战场布局**当数据画**进一个 N×N 网格，导出 `demo_map.json`。

- 顶部是一堆**布局数据块**：`RIVER_CENTERS`（每行河中心列）、`FORESTS`、`HOUSES`、`ROADS`、`BRIDGES`、`WALLS`、`CLIFFS`，全是坐标。
- 用 `paint_rect` / `paint_river` / `paint_road` 这些小函数把特征"刷"进网格。
- 有一个 `PALETTE` 调色板，定义每种地形的 4 个属性：
  - `rgb`：语义分割图的颜色
  - `walkable`：地面单位能不能进/停
  - `move_cost`：进入消耗的移动力
  - `height`：深度图用的灰度高度
- 跑完打印一张 **ASCII 预览**，直接在终端肉眼看布局对不对。

**为什么这么设计**：想改地图，只改顶部坐标数字即可，**不用手敲 625/1225 个格子**。这一份 JSON 就是布局的唯一真相。

### 3.2 `render_map.py` —— 把 JSON 渲染成控制图（"数据翻译成图"）

**它干什么**：读 `demo_map.json`，输出到 `scripts/map/out/`：

| 文件 | 用途 |
|---|---|
| `seg.png` | 语义分割（彩色硬色块），喂 ControlNet 的 **segment** 类型 |
| `seg_soft.png` | 高斯模糊版 seg（给高 denoise img2img 实验用，后被淘汰） |
| `depth.png` | 灰度高度图，喂 ControlNet 的 **depth** 类型 |
| `canny.png` | 边缘线图，喂 canny（后证明不适合，淘汰） |
| `walkable.png` | 黑白可行走遮罩，进 Unity 逻辑层 + 调试 |
| `preview.png` | seg + 网格线 + 图例，**纯人眼预览** |

**为什么要出这么多张**：一开始不知道哪种控制图最好用，全出出来逐个试。最后定论见第五节。

## 四、地图布局的迭代（数据驱动的好处）

一路改了好几版，每次都只是改 `build_demo_map.py` 顶部数字、重跑两条命令：

1. **25×25 初版**：S 形河 + 两桥 + 两岸村庄 + 四角森林/悬崖。
2. **20×20**：用户觉得 25 格子太多，降到 20，更贴参考图密度。
3. **35×35**：用户指出参考图其实是 40×40 级别，放大重铺一版。
4. **房子放大**：原本每栋房子只占 1 格，"人和房一样大"不行 → 改成右上两栋 3×3、左下两栋 3×2 的大房子，并按用户要求**错落摆放、间隔 3 格**。

**核心体会**：格子数其实**不影响 Flux 最终画质**（控制图都是同尺寸色块），只影响**游戏逻辑精度**。35×35 = 1225 格对 SRPG 一关偏大，后续可能回调。

## 五、出图方法的探索：img2img 失败 → ControlNet 成功（关键经验）

这是今天最值钱的部分。Flux 跑在本地 ComfyUI（`127.0.0.1:8188`，另一个仓库 `FamilyArtGeneration`，**agent 不碰**）。模型是 `flux1-dev-fp8`（5070Ti 能跑的最好档）。

### 5.1 先试 img2img（把控制图当初始图）——**失败**

把 `seg.png` 当 init image，调 denoise（去噪强度）：

```
denoise 低 (0.55) → 死守色块 → 出来是 Minecraft 像素块
denoise 高 (0.72) → 够 painterly → 但还是块状 / 或布局跑飞
```

**根因**：img2img 用**同一个 denoise 旋钮**同时控制"守布局"和"出画风"，**必然顾此失彼**。
- 试过 GPT 建议的"模糊 seg（`seg_soft.png`）+ 高 denoise 0.8"：画风是 painterly 了，但**图也跟着糊**（起点糊 → 输出糊，UltraSharp 也救不回）。

**结论**：img2img 这条路在"锐利 + 守布局"上有**结构性天花板**，放弃。

### 5.2 改用 ControlNet——**成功**

下载 `FLUX.1-dev-ControlNet-Union-Pro-2.0`（一个 6.6GB 文件，canny/depth/seg 全包）。

ControlNet 的本质优势：**把"结构"和"像素"拆成两个独立旋钮**。

| | img2img | ControlNet |
|---|---|---|
| 结构来源 | init 图（硬色块→块状） | 独立控制图 |
| 出图起点 | 从色块图"擦"出来 | 从**纯噪声**开始 |
| denoise | 被迫两难 | **拉满 1.0** |
| 画风 | 被起点拖累 | 完全 painterly（full denoise 天然锐利） |
| 布局 | 跟着 denoise 飞 | 控制图死死摁住 |

**连法**（在已有的带 3 个 HD-2D LoRA 的 srpg workflow 上加节点）：
```
Load ControlNet(Union-Pro-2.0) → SetUnionControlNetType → ControlNetApplyAdvanced
Load Image(seg.png 或 depth.png) ─────────────────────────┘
KSampler: denoise 1.0, cfg 1.0, steps 25, EmptyLatentImage 1024×1024
```

### 5.3 踩过的坑 & 调参结论（务必记住）

1. **`SetUnionControlNetType` 必须和控制图类型匹配**！
   - 喂 `seg.png` → type 设 `segment`
   - 喂 `depth.png` → type 设 `depth`
   - **曾经把 depth 图配着 `segment` 类型跑，出来一团乱**，差点冤枉 depth 不行。切对类型才是公平测试。

2. **seg + strength 0.7 = 最佳平衡**（正俯视、布局守住、房子干净）。
   - strength 0.5：更美但**视角斜了**（低 strength 时 tilt-shift LoRA 挣脱约束，按等距视角画）
   - strength 0.9：发灰扁平（seg 硬色块直接糊上来盖过画风）
   - canny：房子鬼影太多，淘汰

3. **depth（类型切对后）**：地形质感更自然，**但房子扭曲、桥乱**（depth 读的是高低不是布局，把房子当"凸台"、把桥和地面混同），还会把平原**拱成一座岛**。
   - 最佳用法是 **seg(0.6) + depth(0.3–0.4) 双 ControlNet 叠加**——seg 管"东西在哪"、depth 加一点地形起伏。demo 阶段没必要，留作精修。

4. **`start_percent` / `end_percent`** = ControlNet 在采样步数的哪一段生效。默认 `0.0→0.8`（前 80% 摁布局，后 20% 放手让模型自由收尾出锐利细节）。`end↑0.9` 布局更稳、`end↓0.6` 画面更自由不僵。

5. **Flux 在 CFG=1 时负向 prompt 完全无效**（guidance distillation 特性）。要排除的东西用**正向肯定句**写（如 `clean empty terrain, no grid lines`）。

6. **尺寸**：Flux 甜区约 1MP（1024×1024），别直出 1440+（易崩）。出图后再 **4x-UltraSharp 放大 → 缩到 2240**。

## 六、关于"房子"的重要架构结论（影响后续所有关卡）

讨论到城市战、室内战时，明确了一个**分层框架**：

```
逻辑层（JSON，永远同源）       ← 不变
美术层 = 底图 + 物件 sprite     ← 按场景换策略
```

| 场景 | 底图 | 物件层 |
|---|---|---|
| 野外/村庄 | AI 出 painterly 地形 | 房子/树/桥用 sprite |
| 城市 | AI 出石板/街道底图 | **建筑模块 sprite**（墙段/屋/门/塔，可重复拼） |
| 室内 | 平铺 tile 地板 | 墙/柱/家具 sprite + Y-sort 遮挡 |

**核心原则**：
- **底图**（连续、不可交互的地面）→ AI 或 tile
- **物件**（独立、可遮挡、可点击）→ **sprite，建库跨关复用**（和角色一个哲学）
- **逻辑** → JSON，永远同源

**只有"纯背景、永不交互、永不遮挡"的远景**才值得让 AI 画进底图。凡是角色会站、会被挡、会点击的物件，都该是 sprite。

> 这也解释了为什么"AI 把密集房子画糊"是死结——**本就不该让 AI 一次画进背景**。
