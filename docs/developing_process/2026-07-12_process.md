## 0. 起点：Seedream 5.0 Pro 让地图产线成立（2026-07-11）

- **结论钉死**：SRPG 地图 = **dual-grid 锁布局/对齐/walkability（JSON 为唯一真源）+ Seedream 5.0 Pro 出整图观感**。
- 刚上线的 **Seedream 5.0 Pro** 重绘是**像素级压格**的：开 gizmos 验证，水面波纹 + 所有 tile 严丝合缝落在格子里，**零偏移**。GPT Image 一直有轻微漂移 = SRPG 致命伤。
- 最省的输入→产出：dual-grid 布局渲染 + **一张 GPT 画的边缘渐变 sheet**（锁接壤风格）→ Seedream 5.0 Pro 整图重绘 → 压回格子 = **成品级**（对比旧的纯色塑料 tile 板天壤之别）。
- 交付机制：`DualGridBuilder` 里的 `seedream_ground_overlay`（sortingOrder -4981），JSON/walkability/objects 全不动。
- 对比结论（用户四图对比）：观感差距约 8/10，但 Seedream 是"成品皮"、dual-grid 是"可编辑骨架"，二者不是二选一。

## 1. 房子/树/桥 = 独立 object sprite（不 bake 进 Seedream）

**决策**：object 用 PixelLab 像素件，摆在 Seedream HD 地面上。人物是像素，物件也是像素，不违和。**不把 object bake 进 Seedream 整图**，理由：

1. **保留游戏功能**：房子是独立 sprite 才能做 YSort 遮挡（人走到房子后面被挡）、单独挪位/选中/交互。bake 进地面图这些全丢。
2. **精准重绘会崩**：Seedream 之所以零偏移是因为喂的是近 segmentation 的平色块；房子/树没法用 tile 表示，喂回去会被挪位/改大小。
3. **迭代成本**：object 摆放最需反复微调，bake = 每次改都重跑 API。

三层「存在方式」：
| 层 | 形态 | 编辑期可见? |
|---|---|---|
| 地面 | Seedream 一张图（真 bake） | ✅ |
| 房子/树/桥 | 独立 sprite GameObject | ✅ 各自独立 |
| 小人 | BattleRunner 运行时 spawn | ❌ 只在 Play |

## 2. NEW：object 摆放往返工具 `ObjectPlacementIO.cs`

**痛点**：用户手动在 scene 里摆 object（房子/树/桥/农田/石头），但 builder 不知道它们在哪；截图靠肉眼估坐标不准。

**方案**：像 `MapExporter` 一样做「场景 → JSON → 重建」往返，直接读 `transform.position`（像素级精确）。

`Assets/Scripts/Editor/ObjectPlacementIO.cs` 菜单：
- **Export Placed Objects**：扫场景里所有 sprite 路径以 `Assets/Art/Objects/` 开头、且不在 `DualGrid/GroundTiles/MapGrid` 下的 SpriteRenderer，写入 `Assets/Art/Maps/stage1_objects.json`（记录 name/sprite 路径/世界坐标/缩放/flip/sortingOrder/YSort/MapObstacle 尺寸）。
- **Build Placed Objects**：读 JSON，清掉散落旧件 + 旧 `ScatteredProps`/`PlacedObjects` 容器，全部重建到一个 `PlacedObjects` 容器（去重 + 整理）。
- **Snap Placed Objects**：x/y 吸到最近 0.5 格 + 给没有 YSort 的补上。
- **Rename Placed Objects**：按类型整齐重命名（`tree_1`/`house_1`/`bridge_1`…），去掉 Unity 的 ` (1)` 复制后缀（`RenameByType=false` 可切纯 `object_N`）。
- **Clear Edge-Clip (restore hidden objects)**：急救，撤销遮罩裁切（见 §4）。

`BattleSceneBuilder`：若 `stage1_objects.json` 存在 → `ObjectPlacementIO.Build()`（跳过程序化撒树 `PropScatterTool.Scatter`）；否则退回撒树。CleanupPrevious 加了 `PlacedObjects` + `MapClipMask`。

**决策**：object 是**编辑期可见的 GameObject**（跟 DualGrid/props 一类），**不像人物那样运行时 spawn**——因为 object 是静态布景+障碍，要肉眼摆位/看 YSort/调美术。人物运行时 spawn 是因为它是模拟实体。JSON 为唯一真源，`PlacedObjects`/`DualGrid`/`ScatteredProps` 的 scene churn **不 commit**（revert 掉）。

## 3. YSort 升级：编辑期实时 + X 次级排序

`Assets/Scripts/View/YSort.cs`：
- 加 **`[ExecuteAlways]`**：编辑期不进 Play 也实时排序（摆 object 当场看正确叠放）。
- 加 **X 次级排序** `xTieBreak`（默认 1）：同一行（foot Y 相等）时，**X 小的画在前**。
- 公式：`sortingOrder = layerBias + round(-footY*unitsToOrder - footX*xTieBreak)`。`xTieBreak << unitsToOrder(100)`，X 只在同 Y 行内决胜负，绝不跨 Y 层。
- 房子/树 pivot 都是 `alignment: 7`（BottomCenter），按脚底排序，一致。

## 4. 坑一：地图边缘裁切（做了又撤——推迟到相机系统）

**过程**：一开始给 object 加了 SpriteMask 矩形遮罩，把探出地图边缘的部分裁掉（`EnableEdgeClip`）。用户随后想清楚了：**这事该推迟**。

**业界实证（用户要求搜，不猜）**：Unity 官方 Cinemachine Confiner 2D —「confine camera position so the screen edges stay within a 2D polygon」= FE / 梦幻模拟战手游 的标准。**相机一旦 confine，视口永不越过地图边缘 → 物理上看不到虚空/探出的物件 → 成品里 object 裁切根本不需要**。

三种边界风格：(1) confine 相机 + 外圈标 not-walkable（FE/梦战手游，最适合我们平俯视 HD）；(2) 装饰边框 + 黑底（超级机器人大战/老梦战）；(3) 纯黑虚空（FFT，但那是 45° 等距+悬浮地块，平俯视露黑边会显没做完）。

**决策**：边界 = 后期 polish（做相机系统时一次性解决），**不在 graybox 阶段做**。裁切代码保留但 `EnableEdgeClip=false`（一行开关，将来若选纯黑边风格再开）。

**关键澄清**：Scene 视图永远看得到出圈物件（编辑器上帝视角）；Play 里相机 confine 后看不到。二者不冲突。

## 5. 坑二：SpriteMask 把房子隐藏了（自愈修复）

旧版 `Snap + Clip` 给**所有** object（含房子）设了 `VisibleInsideMask` + 建了 `MapClipMask`。但 **SpriteMask 在本项目 URP 2D 里对这些渲染不可靠**（仓库记忆早记过它对 TilemapRenderer 没用）→ 房子被遮成**不可见**，用户点树的位置选中的其实是看不见的房子（scale 1.5 是房子真实尺寸，正常）。

**修复**：`Snap` 在裁切关闭时**主动把 maskInteraction 重置为 None + 删 MapClipMask**（自愈）；另加 `Clear Edge-Clip` 急救菜单。跑一次房子全回来。

**教训**：关掉功能开关不会撤销已经写在物体上的状态；要么主动重置，要么给急救菜单。

## 6. 坑三 = 真 root bug：Transparency Sort Mode 没配成 2D（Z-fighting）

**症状**（用户："随着 zoom 所有东西都变成了不一样的 object"）：重叠 sprite 缩放时来回翻转谁在前面。

**排查**：无 Sprite Atlas（排除图集/mip）→ 锁定 sortingOrder 打平后的 **Z-fighting**。读 `ProjectSettings/GraphicsSettings.asset`：`m_TransparencySortMode: 0`（Default）+ 轴 `{0,0,1}`（按 Z/距离）。平俯视 2D 全在 z=0 → 距离相等 → 排序打平时按距离决胜负 = 不稳定 → 缩放/批处理翻转。`[ExecuteAlways]` YSort 只是把这个隐藏问题在编辑期暴露了。

**修复**：`m_TransparencySortMode: 3`（CustomAxis）+ `m_TransparencySortAxis: {x:0,y:1,z:0}`（标准 Unity 2D 设置；2D 模板自带，本项目迁移过来没配）。tied sortingOrder 改按**世界 Y**决胜负（低 Y 在前）= 稳定+正确，且不动现有 sortingOrder 波段（地面 -5000 < object < 人物 10000）。GraphicsSettings 改动需 Unity 失焦/重启才读到。

**教训**：`const bool` 开关会触发 `CS0162 unreachable code`（编译器当编译期常量把 else/if 分支判死）→ 改 `static readonly` 消除警告且保留开关。

## 8. PixelLab object 生成 API 调研（本次收尾，pipeline 起步）

用户给的示例是**网站 Object Creator**（`pixellab.ai/create-object/{id}`）：
- 房子 `0990630e-f8bf-45bb-958d-198185841613`
- 桥 `76900c10-2642-474b-8298-02ac0dd0d8c4`

**两套 object 端点要分清（成本差 20-40 倍！）**：

| 端点 | 用途 | 尺寸 | 成本 | 备注 |
|---|---|---|---|---|
| **`POST /v2/map-objects`** | 透明 top-down 地图物件（树/房子/桥/道具） | **32–400px 任意宽高**；basic 模式 ≤400×400 面积；inpainting 模式 ≤192×192 | **~1 generation**（便宜） | `scripts/pixellab_map_object.py` 走这个 = **产线主力** |
| `POST /v2/create-1-direction-object` | 单方向 object（网站 create-object 就是它） | `size` 32–256（方形），默认 64 | **20–40 generations**（贵！） | 会出多个候选进 review 选帧；用户示例 URL 是这套 |
| `POST /v2/create-8-direction-object` | 8 方向 object | `size` 32–168 | 20–40 gen | 需转向物件时才用 |

**`/v2/map-objects`（CreateMapObjectRequest）可调参数**：
- `description`（必填，≤2000 字）
- `image_size`（默认 128×128；**32–400px 任意宽高比**；常用 64/128/192；basic ≤400×400 面积、inpainting ≤192×192）
- `view`：`high top-down`(默认,~35° 俯角) / `low top-down`(~20° RPG 视角) / `side`
- `outline`：`single color outline`(默认) / `selective outline` / `lineless`
- `shading`：`flat` / `basic` / `medium`(默认) / `detailed`
- `detail`：`low` / `medium`(默认) / `high`
- `text_guidance_scale`：1–20（默认 8）
- `init_image` + `init_image_strength`(1–999,默认 300)：参考初始图
- `color_image`：强制调色板
- `background_image` + `inpainting`(mask/oval/rectangle)：把物件贴进指定地图做风格匹配
- `seed`
- 异步：返回 `background_job_id` + `object_id`，轮询 `GET /v2/background-jobs/{job_id}`；完成后 `GET /v2/objects/{object_id}` 下载。

**结论/pipeline 方向**：object 产线走 **`/v2/map-objects`**（便宜 1 gen、支持大尺寸+任意宽高+init/背景匹配），网站 Object Creator（贵）只在需要 review 候选/多方向时用。

**prompt 修订原则**（用户提）：
1. **去掉 "16-bit / GBA" 字眼**（这些让它偏老式/掉细节）。
2. **房子不要额外草地**：好的示例没草地，但生成不稳定（多数带草地）。修订 prompt 强调「isolated building only, no surrounding grass/ground/terrain patch, transparent background」——待验证是否稳定（类比 SuYao "hood" 教训，负向词有时反而触发，需实测）。

新建 `pixellab/objects/{houses,bridges,trees}/` 目录 + `pixellab/object_status.yaml`（镜像 `character_status.yaml`，记录每个 object 的 endpoint/size/view/params/prompt/状态）。
