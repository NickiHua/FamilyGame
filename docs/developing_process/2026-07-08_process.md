## 二、火山 Visual 调用方式：Ark Key vs AK/SK

这次把火山体系分清了。

### 2.1 Ark / Seedream

用于 Seedream 5.0 / 图像生成：

```text
Endpoint: https://ark.cn-beijing.volces.com/api/v3/images/generations
Auth: Authorization: Bearer <Ark API Key>
本地 key: volcenginekey.txt
```

模型列表查询确认账号可用模型包括：

```text
doubao-seedream-3-0-t2i-250415
doubao-seedream-4-0-250828
doubao-seedream-4-5-251128
doubao-seedream-5-0-260128
doubao-seedream-5-0-pro-260628
```

文档标题写 `Seedream 5.0 lite`，但账号模型里没有单独 `lite` 后缀；本次使用 `doubao-seedream-5-0-260128` 作为非 pro / lite-class 档。

参考图字段确认：

```json
{
  "image": "data:image/png;base64,..."
}
```

### 2.2 Visual / CV / 即梦 / 旧 inpaint / saliency

用于火山视觉服务、旧 inpaint、即梦交互编辑、通用分割：

```text
Endpoint: https://visual.volcengineapi.com
Action: CVProcess 或 CVSync2AsyncSubmitTask / CVSync2AsyncGetResult
Version: 2022-08-31
Region: cn-north-1
Service: cv
Auth: AK/SK V4 HMAC 签名
本地 key: volcengineak.txt + volcenginesk.txt
```

签名要点：

```text
SignedHeaders = content-type;host;x-content-sha256;x-date
X-Date = UTC YYYYMMDDTHHMMSSZ
X-Content-Sha256 = body sha256 hex
Authorization = HMAC-SHA256 Credential=AK/Date/Region/Service/request, SignedHeaders=..., Signature=...
```

脚本里用标准库实现了签名，并用官方签名示例验证过 HMAC 链路：最终 signature 能复现官方 POST 示例。

## 三、实验 1：火山主体分割 / 抠图 saliency_seg

### 3.1 文档结论

主体分割接口：

```text
Action: CVProcess
req_key: saliency_seg
binary_data_base64: [原图]
only_mask: 3 或 4
rgb: [-1, -1, -1]
refine_mask: 0/1/2
```

关键参数：

```text
only_mask=3: 返回原图大小的 BGRA 透明前景图
only_mask=4: 返回原图大小 BGRA 透明图 + 原图大小 mask 两个产物
rgb=[-1,-1,-1]: 返回透明底，而不是白底 / 彩底
return_url=true: 文档写 URL 输出默认 jpeg，不支持 alpha，所以要避免
```

### 3.2 实测

输入：

```text
art_undecided/portraits/luli/luli_full_2k_transparent_doubao-seedream-5-0-260128_20260707_225710.png
```

输出：

```text
art_undecided/portraits/luli/luli_full_2k_transparent_doubao-seedream-5-0-260128_20260707_225710_volc_saliency_seg.png
```

验证结果：

```text
size: 2048x2048
mode: RGBA
alpha extrema: (0, 255)
alpha bbox: (635, 89, 1558, 1959)
```

### 3.3 质量判断

优点：

- 真 alpha PNG，通道上没问题。
- 主体完整抠出来了。

缺点：

- 边缘羽化 / 软边偏严重。
- 发丝、披风、盔甲边缘不如当前 `Real-ESRGAN x4 + 手工抠图 + 缩回` 工作流。

结论：

```text
火山 saliency_seg 可作为快速预览 / 粗抠图工具。
不适合作为最终立绘透明资产工作流的上限。
当前 X4 + 手工抠图仍然更强。
```

## 四、实验 2：旧 Visual inpainting 编辑（下线中）

### 4.1 文档结论

旧 prompt inpainting 接口：

```text
Action: CVSync2AsyncSubmitTask / CVSync2AsyncGetResult
req_key: i2i_inpainting_edit
binary_data_base64: [RGB 原图, 灰度 mask]
custom_prompt: 文本提示
scale: 1-20，默认 5
steps: 默认 25
seed: 默认 -1
```

mask 规则：

```text
黑色 0 = 保持
白色 255 = 重绘
```

文档标题明确标记：

```text
inpainting 涂抹编辑（下线中）
```

### 4.2 第一轮：整脸 mask + 英文 prompt

输入：

```text
art_undecided/portraits/luli/luli_full_2k_transparent_doubao-seedream-5-0-260128_20260707_225710.png
```

输出：

```text
art_undecided/portraits/luli/luli_full_2k_transparent_doubao-seedream-5-0-260128_20260707_225710_volc_inpaint_angry_face.jpg
```

结果：

- 接口成功，返回 JPG / RGB，无 alpha。
- 表情确实变“生气”。
- 但脸部漂移严重：像长了面罩 / 护脸，眼睛变黄，角色身份被破坏。

判断：

```text
灾难级，不能作为角色表情编辑工具。
问题同时来自旧模型、mask 太大、prompt 过度发散。
```

### 4.3 第二轮：新图 + 小 mask + 中文 prompt + 原始 JPG 输入

输入：

```text
art_undecided/portraits/luli/luli_full_2k_transparent_doubao-seedream-5-0-260128_20260707_225931.png
```

策略：

```text
input-encoding: jpg
mask: 只罩眉眼和嘴巴三块，不罩整脸
feather: 3
scale: 2.5 / 3.0 / 3.5
prompt: 中文，强约束蓝眼睛、禁止面罩、只改眉眼嘴
```

输出：

```text
art_undecided/portraits/luli/*_volc_inpaint_angry_face_zh_scale25.jpg
art_undecided/portraits/luli/*_volc_inpaint_angry_face_zh_scale30.jpg
art_undecided/portraits/luli/*_volc_inpaint_angry_face_zh_scale35.jpg
```

结果：

- 没有再长面罩。
- 蓝眼睛保持住了。
- 表情可读性提升。
- 但仍有脸部润色 / 眼睛风格被重画的味道。

结论：

```text
旧 inpaint 在小 mask + 中文 prompt + 低 scale 下能做表情草稿。
但不适合作为最终立绘表情生产线。
```

主要观察：

- 即梦在裁剪立绘上局部性强很多。
- 表情幅度大，默认语义惯性强：`高兴` 容易露齿笑，`难过` 容易掉眼泪。
- 否定词效果不稳：`不要露牙` / `不要眼泪` 仍可能失败。
- 正向描述更好：
  - `闭嘴微笑，嘴唇合拢，嘴角小幅上扬`

- `低落 / 遗憾` 比 `难过` 更适合克制剧情立绘。
- `thin_3` 是较可用的克制微笑候选。

结论：

```text

适合做表情候选，但仍需 GIF 检查和人工修嘴/眼。
```

## 七、实验 5：SeedEdit3.0 / SAMTok 调研

### 7.1 SeedEdit3.0

发现新接口：

```text
图生图3.0-指令编辑 SeedEdit3.0
docs/86081/1804561
req_key: seededit_v3.0
```

参数：

```text
输入 1 张图
prompt 必填
seed 可选
scale: 0-1，默认 0.5，越大越听文字、越少受原图影响
无 mask 参数
```

判断：

```text
SeedEdit3.0 可能比旧 inpaint 强。
但不支持 mask，对角色表情/局部资产而言不是主力。
可作为备选图生图编辑工具。
```

### 7.2 SAMTok

查到：

```text
SAMTok: Representing Any Mask with Two Words
CVPR 2026 Highlight
```

本质：

```text
把任意 mask 压缩成两个特殊 token，让 QwenVL 等 MLLM 理解/生成 mask。
底层基于 SAM2。
```

判断：

```text
SAMTok 不是 rembg / 抠图模型的直接替代。
它解决的是“多模态模型如何表示和生成 mask”，不是发丝级 alpha matting。
最多适合作为未来交互式选区工具方向。
```

## 八、实验 6：local ComfyUI + Illustrious XL + FaceDetailer 表情编辑

用户在本地 ComfyUI 中搭出工作流：

```text
LoadImage 原全身图
CheckpointLoader: Illustrious-XL
正向提示词: 身份锚点 + 表情
负向提示词
YOLO 脸检测
SAM 精细分割
FaceDetailer
SaveImage
```

结果：

- FaceDetailer 只处理脸部 crop / mask，局部性强。
- 能在不污染身体、服装、披风的情况下改变表情。
- 比云端旧 inpaint 稳定，比即梦更适合本地批量抽表情。

重要理解：

```text
FaceDetailer 不是“只能画脸”，而是“处理 detector 找到的区域”。
接 face detector 就修脸；接 hand detector 就修手；接自定义 bbox/mask 理论上能修任意区域。
```

生产判断：

```text
表情差分最怕污染头发、衣服、身体、角色轮廓。
FaceDetailer 只动脸，反而是优点。
```

建议定位：

```text
表情资产主流程：local Comfy FaceDetailer + 人工挑选 + Photopea/PS 点修
云端即梦：备选 / 快速参考
旧 inpaint：淘汰
```

## 十六、重要教训

1. **看图不能只看“好看”。** 资产要看结构一致性。GPT 图完成度强，但服装结构漂，游戏里不可直接混用。
2. **Mask 是局部编辑成败核心。** 不支持 mask 的工具只能做参考，不能做主力局部编辑。
3. **否定 prompt 不稳定。** `不要露牙` / `不要眼泪` 可能仍失败；正向写 `闭唇微笑` / `脸颊干净` 更好。
4. **表情 prompt 不能写“保持五官完全不变”。** 更准确是：保持身份 / 轮廓 / 颜色；只允许眉眼嘴小幅变化。
5. **Comfy FaceDetailer 是表情差分最接近生产线的本地方案。** 它把 blast radius 控制在脸部。
6. **Seedream 5.0 参考图一致性很强。** 适合角色动作 / 武器 / 姿势差分。
7. **PixelLab sprite 一致性粒度不能按高清立绘要求。** 玩家认身份锚点，不认腰带扣具体形状。
8. **最终 5% 仍是人工。** 眼睛大小、眉毛粗细、嘴线、边缘、alpha，需要 Photopea/PS/Aseprite 修。

### 17.1 BytePlus / 火山入口差异

最初误区是把以下入口当成同一套能力：


### 17.8 Ark SeedEdit / mask inpaint 调研
