## 二、角色「头消失」—— 是**层级**问题，修为 YSort `layerBias`（层偏置）

- 现象：角色的头被**房子/物件盖住**——本质是**排序**（谁画在前）问题。
- 修复：给 `YSort.cs` 加 **`layerBias`**（`sortingOrder = layerBias + round(-footY*unitsToOrder)`）：
  - **角色**（`BattleRunner` 出生的单位）`layerBias = 10000`（`UnitLayerBias`）→ **永远画在房子/物件之上**，头再不会被盖。
  - **房子** `layerBias = 0`。
  - 角色**之间**仍按脚底 Y 互相遮挡（靠下的在前）——斧兵/弓箭手上下叠放谁前谁后正确。
- 三层分清：**地面/水/桥（-49xx/-5000）最底 → 房子（0~-3000）中间 → 角色（+7000~10000）最上**。

## 三、决策：重铺地图为程序化 dual-grid

放弃「手摆 base tile」，改由 `DualGridBuilder.cs` 从 `stage1_map.json` 程序化生成整张地面：

- **底 ground**（order -5000）：每格一张平铺 base 瓦。
- **3 层 dual EDGE**（升序叠加）：dirt / road / grass，各用**同一套 16 张 dual 瓦**（`{terrain}_dual/00-15.png`，由 3 张手绘 mask 经 `scripts/tiles/gen_dual_tiles.py` 填不同纹理生成）。
- **水切 4 块做底**：水**不做 dual 层**，只做 base——把一张 128px 水图切成 `water_base_1..4`，底图按 `water[(x&1)+2*(y&1)]` 的 **2×2 pattern** 铺，让大片水不那么重复。水边由更高地形的 dual 压过来形成。
- **桥作为独立层/object**：`'D'` 格在最上面单铺 bridge（order -4980），角色仍 Y-sort 在其上。
- **优先级压盖**：dual 层 `On(corner)=Priority>=layerPrio`，高优先地形的 dual 画在低的之上、往低的方向探 delta 半格。各层用**相同瓦形**，所以高的正好盖住低的、无缝隙。
- **MapExporter 配套修**：①`water*` 前缀名 → 'W'（修 water_base_* 被当成 grass 导出）；②`MapObstacle` 格盖 'B'；③新菜单 `Tools/Map/Clean Stage1`＝删掉「有 tile 但 sprite 为 null」的坏水瓦（`water.png` 删除后残留、仍占格 + 导出成 'W' 把地图撑到 77×31）→ 清成 **30×30**。

> 无缝规则（已验证）：grass 在每条混合边都伸到 **42px**（越过中线 32 的 delta=10 探入空侧），保证 16 张瓦两两接缝一致。

## 四、铺 dual-grid 的六连坑（本 session 重头，用户逐个指认，我逐个修）

> 总教训：**用户从头到尾每个都点对了**；我几次「嘴硬/误判」，最后靠**客观测量**（量瓦四象限 alpha 占比、量边缘像素偏移、按坐标渲染放大）才跟上。

**坑 0（Issue 1，先修好的）——瓦戳出地图边**：dual 网格天生比 base 大一圈，最外圈会探出半格。当时用 **interior-only 循环**（`x∈[0,w-2],y∈[0,h-2]`）跳过外圈 → 干净矩形边。（后来发现这会害边界不 blend，见坑 4。）

**坑 1——内部 3 地形交界出「直角台阶」**：用户圈了路(col27)/row23 处。我一开始误判「无 bug」，放大后确认是真的。
- 根因 = **优先级翻转台阶**：竖直的路两侧邻居从「上面草(吃路,+11px)」变成「下面土(被路吃,−11px)」，路缘方向一翻转就**跳 2×delta=22px** 硬台阶（实测 +11↔−11）。
- 修复：**把路降为最低非水优先级**（`水0 < 路1 < 土2 < 草3`）。这样路永远被吃、边缘恒 +11 一致、不再跳。代价是土/草会有机地「啃」进路边一点（看着更像村道）。
- 通则：优先级压盖式 hybrid 里，**直边在「邻居相对优先级翻转」处会跳 2×delta**。把「穿场」地形（路）放最低，让它永远被吃，就不会翻转。

**坑 2——水面凸起「顶圆底方」= 瓦 b11/b13 装反了**：用户揪住河湾一个 1 格凸起「顶角圆、底角是直角」。我先误判成「1 格河湾几何」、还说「水做 dual 没用」，都错。
- 客观测量（每张瓦四象限不透明占比 vs bits 应有象限）**铁证**：`gen_dual_tiles.py` 里 `three`(3-corner) 的旋转映射把 **11 和 13 的缺口写反了**（PIL `ROTATE_90` 是逆时针：缺口 BR→TR；`ROTATE_270`→BL）。生成的 b11 缺口在 TR（应 BL）、b13 缺口在 BL（应 TR）。凸起顶角用 b7(对)、底角用 b13(错) → 顶圆底方。
- 修复：`13: rot(three,R90), 11: rot(three,R270)`（互换）。corner(1/2/4/8)、half(3/5/10/12)、7/14 都是对的，只有 11/13 反。重生成三套，量化复验 4 张 3-corner 缺口全部归位。

**坑 3——地图边界地形变化处是硬直边**：坑 0 的 interior-only 副作用——最外圈整个被跳过，边缘上的地形变化（草/水、草/路）不 blend、成硬直切。
- 修复：dual 循环改成**全外圈** `for y=-1..h-1, x=-1..w-1`，并把**出界采样 clamp 到最近边缘格**（`Mathf.Clamp`）。纯色边采样全同 → bits 0/15 跳过（不冒假地形）；只有边界上真有变化处才 blend。

**坑 4——边界 blend 瓦探出地图矩形「留尾巴」**：外圈 blend 瓦本质探出半格到 void。
- 先试 **SpriteMask**（矩形遮罩裁到地图矩形）——**Unity 里对 Tilemap 没裁住**（"没起作用"），且我没法本地验证，**弃用**。
- 改为**裁边瓦（clipped border tiles）**：`gen_dual_tiles.py` 额外产出**每地形 8 张裁半格变体**——clamp 后边界只会用到竖边 `3/12`、横边 `5/10`，所以出 `03L 03R 12L 12R`(左右边抹外半) + `05T 05B 10T 10B`(上下边)。builder 外圈按边选变体。**烘进资源、Python 所见即所得**，不靠运行时遮罩。

**坑 5——裁边瓦 Y 朝向镜像了**：用户「(29,1) 草底半格该切没切、高级那块整块探出」。
- 根因 = **我的验证图 y 向下、Unity y 向上**。同一循环下标 `y==-1` 在 Python 是**顶边**、在 Unity 是**底边**（`TerrainAt=base[H-1-y]` 翻转 + Unity 屏幕 y=0 在底）。我按 Python 写了 `y==-1→"T"`，Unity 里底边被当顶边裁 → 探出的底半格留着、内侧反被抹。
- 修复：Unity 里 `y==-1→"B"`、`y==h-1→"T"`（X 不受影响，左右一致）。内部瓦其实 Python/Unity 一致（都 NW 角→PNG 左上），只有「边→裁向」这层镜像。

**坑 6——裁边 PNG 没被 Unity 导入**：那 8 张 PNG 是外部生成的，Unity 不自动吃；我上次还误说「y 改动不用 reimport」坑了用户 → `GetOrCreateTile` 加载不到 → 回退整张瓦 → 尾巴又回来。
- 修复：`GetOrCreateTile` 加载不到且文件存在时 **`AssetDatabase.ImportAsset(..., ForceSynchronousImport)`** 强制导入；`Clip()` 若仍为 null 会 `LogWarning` 报出缺哪张瓦。不依赖用户手动 Reimport。

**顺带——地图重建入口缺失**：`DualGridBuilder.Build` 原本只藏在 `Build Battle Scene` 里，用户「找不到 Build DualGrid」。新增独立菜单 **`Tools/FantacyCentry/Build DualGrid Only`**（解析 JSON、关掉手绘 Tilemap renderer、只重建 DualGrid）。

## 六、经验教训（写给未来的自己）

1. **验证图坐标系要跟 Unity 对齐**：我 Python 渲染 y 向下、Unity y 向上。内部/瓦形看着一致，但**任何「循环下标 ↔ 屏幕边」的逻辑会上下镜像**，边界裁剪就栽在这。以后做边缘相关逻辑，先想清楚这层翻转。
2. **别嘴硬，用客观测量**：视觉看不准就**量数字**——瓦四象限 alpha 占比一量，b11/b13 装反立刻现形；边缘偏移一量，优先级台阶 +11/−11 一目了然。
3. **别上不可验证的 Unity 运行时魔法**：SpriteMask 对 Tilemap 我没法本地验证、Unity 里还不灵。**能烘进资源（裁边 PNG）就烘进去**，Python 渲染即最终效果。
4. **外部生成的资源要 force-import**：脚本写的 PNG，Unity 不自动吃；builder 里 `ForceSynchronousImport` 兜底，别靠口头提醒用户 Reimport。
5. **相似结构渲染不一致 = 有 bug**：用户「同一个凸起顶圆底方」的直觉就是铁律，别用「几何本来如此」搪塞。
