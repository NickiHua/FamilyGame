# 2026-07-22 开发过程日志 · Marker-based 地图重绘实验 + Build Battle Scene V2

> 跨 07-20 ~ 07-23 的一条连续实验线：**把地图上的物件（房子/树/石头/桥）表达成「带字母的纯色 marker tile」，喂给 AI（Seedream / GPT）重绘成 HD 成品地图**。目标是解决「房子门口的路和网格 road 对不齐」的老问题，并顺带对比 Seedream vs GPT、探索「地图只用一张 HD 图 + 逻辑网格」的新架构。

---

## 0. 动机：为什么要 marker

之前的做法是把**真的 object sprite** 摆到地图上。问题：
- 房子 sprite 里**烘死了一截「门口的路桩」**，位置固定，多半不在格子中线。
- 想让「门对齐 road」就得平移房子，一平移**房子左右就不占整格**了，除非拉伸（不想要）。
- 本质矛盾：**刚性 sprite 的门** 没法同时满足「footprint 贴整格」+「门对齐 road」。

**破解思路**：不摆刚性 sprite，改成在网格上画「H marker 方块」。
- **footprint 你画整格 → 天生锁在网格上**（collider 从 JSON 走，不受影响）。
- **门画在哪、怎么接路 → 交给模型现画**（柔性门，自动长到路那一侧）。
- 两个约束各让一步，死结解开。

## 2. Unity 侧工作

### 2.1 复制地图（非破坏）
- `Grid > Stage1` 是手绘编辑画布；`MapGrid` 只是逻辑解析器（读 `stage1_map.json`），**不能画**；`DualGrid` 是从 JSON 现拼的视觉层。
- `Ctrl+D` 复制 `Stage1` → `Stage1_marker`，在副本上铺 marker（照地形参照摆）。
- **注意**：`Stage1` 的 Tilemap Renderer 会被 `Build DualGrid Only` 关掉，所以平时看不见；要编辑先把它勾回来、并隐藏 DualGrid。

### 2.2 地图结构澄清（DualGrid 由哪几层拼）
从下到上：`water_surface(-5010)` + `shore_land(-5009)` + `ground(-5000)` + `road/dirt/grass_edges(-4995~-4985)` + **`stage1_ground_overlay`（SD 出的整图 PNG，读 `Assets/Art/Maps/stage1_b.png`，-4981）** + `bridge`。
- **没有单张「原始 dual-grid 图」**：每次 `Build DualGrid Only` 从 JSON 现拼。真相源头是 `stage1_map.json`，不是任何 PNG。

### 2.3 MapImporter.cs（新，Tools/Map/Import Stage1 JSON to Tilemap）
- `MapExporter` 的**反向**：读 `stage1_map.json` 把选中/`Stage1` 的 Tilemap 重新刷成 JSON 的样子，并把左下角格 snap 到世界原点。
- 起因：`Stage1` 手绘图和 JSON **脱节了**（JSON 的 dirt 在中间、手绘图还在角上）。因为之前只有 Export（tilemap→JSON），没有 Import。
- 复用 `MapGrid.Parse()/RowsTopFirst`，不自己写 parser。
- 字母→tile：`G=grass R=road I=dirt W=water_base(2×2) D=bridge S=sand B=grass`。
- **坑**：不存在 `water.png`，`water.asset` 的 sprite 是空的 → 渲染成深灰；真正的蓝水是 `water_base_1..4`（avgRGB≈40,164,255），按 cell 奇偶铺 2×2。

### 2.4 MarkerMapExporter.cs（新，Tools/Map/Export Marker Map）
- 读**选中的** marker tilemap → 写 `Assets/Art/Maps/stage1_marker_map.json`（**render-only，不碰真 `stage1_map.json`**，walkability 不受影响）。
- Tile 名→字母：地形 `G/R/I/W/D/S` + marker `house(_H)=H tree(_T)=T stone(_S)=K bridge_B=P`（内部用 K/P 避免和 S=sand、B=building、D=bridge-terrain 撞）。
- **坑**：在 tilemap 顶端外面误画了一个孤立 T marker → `CompressBounds` 把网格从 30×30 撑成 30×42。删掉那颗野树、重导即恢复。

## 3. 渲染脚本（marker JSON → 1920 图）

### 3.1 scripts/render_marker_map.py（平铺硬边版）
- `stage1_marker_map.json` → `art/experiments/stage1_marker_render_1920.png`（30×30×64 = 1920²）。
- 地形用平铺 tile（grass/road/dirt/sand，水用 water_base 2×2）；marker cell 填**纯色块**；每个连通 marker 区域用 `scipy.ndimage.label` 画**一个大字母**（内部 K/P 显示成 S/B，因为用户要 H/T/S/B 显式区分——纯色和 dirt 太接近，用户否了纯色版，加字母才分得开）。

### 3.2 scripts/render_marker_dualgrid.py（dual-grid 有机边缘版）
- **Python 完整复刻了 `DualGridBuilder`**：base 地面 + road/dirt/grass 三层升序边缘层、半格偏移、16 配置、边界裁切（`03L/12R/05B/10T` 等 clip 变体）。
- 优先级 `W/D=0 < R=1 < I=2 < G(含 marker 当草)=3`。bit1=up-left…；marker cell 按 grass 处理再盖色块。
- 输出 `art/experiments/stage1_marker_dualgrid_1920.png`：河岸/路缘是波浪状有机过渡，和 Unity DualGrid 一致。以后可复用。

## 6. 房子朝向大战（本 session 最纠结的点）

**问题**：房子总画成**「纵向」= 三角形山墙尖端正对镜头** → 正面又窄又高、几乎全是屋顶、**门显小**。
想要的是**「横向」= 长边屋檐面正对下方** → 正面是宽墙、门开宽墙上、**门大**。

**比例讨论定论**：
- 角色 128px 精灵压到 ~80px ≈ 1.25 格；门 ≈ 人高；
- 「门等高 + 紧凑屋顶」≈ 房子 3 格高，**不必 4×4**（4×4 吃格子太多，30×30 会挤）；
- 参考范式：Tactics Ogre / 圣女战棋 = 门等高的「压扁写实」；火纹 GBA = 房≈人的抽象符号。用户选**门等高、横向**。

**村庄统一**：全村**同一种茅草顶**（颜色深浅可略不同，不要红瓦/灰瓦拼盘），4 栋靠**体量与形状**（大小、方/长条/L 形、门窗、烟囱位置）区分——而不是换材质。

**树分两档**（行业标准）：矮树 `T`（可走，+防御）；大树/森林（阻挡，森林关用，未来加 `F` marker）。所有 `T` 是**同一种矮树**，密集处只是紧挨排列，别画成高森林。

**房子 sheet 生成测试**（house_sheet_prompt.txt 中文 / _en.txt 英文，2×2 四栋）：
- **Seedream 纯文生图** → 出成 **45° isometric**，**画不出正俯视横向**。
- **GPT-image-2 medium** → **横向 + 大门做到了**，但视角是**近正立面**（几乎只看到正面墙、屋顶往后收），**俯视不足**（GPT 画建筑爱塌成 elevation 的老毛病）。
- **结论**：两家各解决一半——Seedream 有俯视没横向，GPT 有横向没俯视。**破法**：给一张**已经是横向俯视的房子图**当 ref（`sample_hourse_31`），让 Seedream 借视角 + 保 layout。

- 农田：木栅栏 + 各种作物。桥：横跨河、两端无缝接路。

## 8. 两家模型的问题（结论）

### Seedream 5.0 Pro
- **优点**：**保 layout 极强**（重绘死守格子，河/房/桥/农田基本贴格），中文 prompt 吃得好。
- **问题**：
  1. **单独生成房子画不出正俯视** —— 纯文生图 = 45° isometric，必须靠一张「已经是正俯视横向」的 ref 才压得住视角。
  2. 整图里房子**朝向跟 ref 走**：没有横向 ref 就默认山墙朝前（纵向、门小）。
  3. 单一强房子 ref → 三栋**克隆雷同**，要靠 prompt + 变化描述才有区别。

### GPT Image 2.0（gpt-image-2）
- **尺寸能力（已查官方文档）**：支持**任意分辨率**——最长边 ≤3840、两边都是 16 的倍数、比 ≤3:1、总像素 655,360~8,294,400；**1920² 合法**（1920=16×120，3,686,400px = 2K 门槛）；常用 2048²/3840×2160。**>2K 标 experimental**。**不支持透明背景**。价格按输出 token（high 1024²≈$0.211，越大越贵，medium 便宜很多）。脚本切换：`--model gpt-image-2 --size 1920x1920 --background opaque`。
- **问题**：
  1. **漂移严重** —— 它是重绘、**没有 Seedream 那种保 layout 能力**，红河/红房块跟画面对不上。**即使 HIGH + 英文强化 prompt（strict template / zero drift / overlay cell-for-cell）也压不住**。
  2. **画面更脏更乱** —— 随机碎石/杂草/过度细节，加了 `SUPPRESS DETAIL` 也只是缓解。
  3. 房子比 SD 稍好、树丛稍好，但整体**不适合保 layout 的地图重绘**。
- **定论**：**GPT-2 只适合出概念/风格图，保 layout 的成品地图还得 Seedream Pro。**

## 11. 水面流动 shader（07-22 ~ 07-23）· flowmap 水 + house5_v1 定稿

> 承接 V2「地图=一张 baked HD 图 + 逻辑网格」的架构。问题：影子/物件都 bake 进了一张平图，**还能不能给水面加流动感？** 答案：能，只要给 shader 一张「哪些像素是水」的 mask。

### 11.1 关键难点
- HD 图是 Seedream 重绘的，河流边缘**已经偏离原始 grid 的 W 格子**（AI 挪柔化过），所以**不能**拿 `stage1_marker_map.json` 的 W 格当水 mask，否则水效果和画面里的蓝水错位。
- 正解：**从 HD 图本身分割**出水 mask。

### 11.2 方案选型（两个独立决定）
- **效果贴哪儿**：A=单图+水 mask（选 A）｜ B=水单独成层 ｜ C=只叠滚动焦散。
- **水纹怎么不露缝**：1=tileable 噪声 ｜ 2=纯程序化噪声 ｜ **3=flowmap 相位混合（选 3）** ｜ 4=双层错速。
- **最终 = A + 3**：单图 + mask 圈定水区；flowmap 逐像素存流向让水**顺河道（从上往下）流**；两个半周期错开样本三角波交叉淡入 → **滚动永不露接缝**，即使噪声本身不可平铺。

### 11.3 数据生成（scripts/water/gen_water_mask_flowmap.py，新）
从 `Assets/Art/Maps/stage1_hd.png` 一次产出两张、写在 HD 图旁边：
- `stage1_hd_watermask.png` — 蓝色色键分割 + 连通域去杂点 + 填洞 + 羽化。water=白。
- `stage1_hd_flowmap.png` — **结构张量**求河道走向（跨河道梯度最强，minor 特征向量=沿河），强制 Y 朝下→水从上往下流，RG 编码方向、水区外中性 0.5。
- 校验：水域 x≈317–779（左三分之一）、贯穿全高、覆盖 5.2%，平均流向 x=0.10/y=0.75（下+略右），符合斜河。

### 11.4 Shader（Assets/Art/Shaders/WaterFlow.shader，新）
URP sprite shader，只处理 mask 标记的水像素：
- **flowmap 相位混合**：`phase0=frac(t)`、`phase1=frac(t+0.5)`、`blend=abs(1-2*phase0)` 三角波交叉淡入 → 无缝滚动、水顺流向。
- **程序化 value noise** 当波纹（无需贴图、天然无限、seam 被相位混合藏掉）。
- 轻微 UV 折射 + 细窄滚动高光（粼光）。房子/影子/草地不受影响。
- **2D SRP Batcher 坑**：`UnityPerMaterial` 里放 `_MainTex_ST` 会报「_TexelSize/_ST 不支持」并禁用 2D SRP 批处理。地图是整张 sprite、UV 就是 0–1，**删掉 `_MainTex_ST`、改用原始 `IN.uv`** 即可。

### 11.5 调参救「肥皂水」
第一版动静太大、像肥皂油花。诊断：**主因是 Refraction（UV 折射把蓝色搅成大块明暗带）**，`Ripple Scale` 在折射归零后几乎无感（它只决定噪声格大小）。真正救回来的是**高光算法重写**：大块低频 crest → 更细更快一层波纹取窄阈值（`smoothstep(0.70,0.95)` + `pow`）= 细碎稀疏亮线。

| 参数 | 旧 | 新 |
|---|---|---|
| Ripple Scale | 14 | 30 |
| Refraction | 0.004 | 0.0015 |
| Ripple Height | 1.2 | 0.5 |
| Flow Advection | 0.35 | 0.22 |
| Flow Speed | 0.08 | 0.05 |
| Sparkle Strength | 0.35 | 0.14 |

> 流向存在 flowmap 贴图里（不是材质滑杆），Inspector 只能调速度/强度，改不了方向；要反向或分段横流得改脚本重生成 flowmap（或后续给 shader 加 Flow Reverse/Angle 参数）。

### 11.6 Unity 接线（BattleSceneBuilder.cs，改）
`Build Battle Scene V2` 自动：创建/复用 `Assets/Art/Materials/WaterFlow.mat`、绑定 mask+flowmap、把两张数据贴图导入设为 **linear/clamp/无压缩/无 mip**（`ConfigureDataTexture`）、挂到 `MapHD`（`EnsureWaterMaterial`）。

### 11.7 草/树风吹（讨论，未实施）
同套路（mask + shader 位移）可做，但 baked 平图两个硬限制：**影子画死不会动**（振幅须极小否则滑动穿帮）、**顶视角草无「根固定梢摆动」结构**（易果冻）。结论：草地收益低暂缓；树/灌木值得做（方案 A 轻摇，或方案 B 抠树冠单独顶点风摇）。
