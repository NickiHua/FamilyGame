# 2026-07-09 开发日志：Stage1 地图水面、岸线与 Seedream 地图增强实验

> 本篇记录一次集中地图美术管线实验。目标是把当前 SRPG demo 地图从「一层 tile 拼图」往《尤特娜英雄战记 / TearRing Saga》那种更整体、更手绘的地形质感推进。重点围绕水面、草水交界、AI 生成图作为整体地形增强层、以及如何在 Unity 里保留原有 walkable / object 逻辑。
>
> 实际工作跨到 07-10，但用户要求记为 0709 日志，因此本文按 2026-07-09 归档。

## 一、背景：为什么要重新看地图美术管线

当前 Stage1 demo 地图已经具备：

- `stage1_map.json` 作为地形语义与 walkable 逻辑来源。
- Unity `MapGrid` 解析 JSON，`G/R/I/S/D/W/B` 等字母决定逻辑地形。
- `DualGridBuilder` 根据 JSON 生成：
  - base ground
  - road / dirt / grass dual-grid edges
  - bridge object
  - props / houses / trees 等 object 仍单独存在
- 水原本是 `water_base_1..4` 的 2x2 tile pattern。

问题是：

- 单层 tile + variants 仍然显得平、规整、重复。
- 水面和陆地硬拼，缺少真正的「岸线」。
- 即使用 dual-grid 平滑边界，水陆交界仍像被 mask 裁出来。
- 用户希望目标接近《尤特娜战记》那种手绘地形整体感。

本次核心原则逐步明确为：

```text
逻辑仍然是 stage1_map.json / grid。
视觉可以是多层整图 overlay / AI 增强图 / 局部 decal。
不要让 AI 决定地图结构，除非它只是作为视觉 overlay 测试。
```

## 二、路线讨论：8x8 水 tile vs 整图 bake 水层

最初讨论过把一张水纹图切成 8x8 tile：

```text
512x512 water source
-> 切成 8 x 8 个 64x64 tile
-> Unity 根据 (x & 7, y & 7) 平铺
```

后来明确：如果只是从同一张 512x512 seamless 水图按坐标重复采样，那么 8x8 tile 和整图 bake 在静态水纹上几乎等价。

差异不在水纹本身，而在承载方式：

```text
8x8 tile: 仍然是 Tilemap 里的碎片。
整图 bake: 一张 1920x1920 透明水层，后续可以共享同一张 mask 做 shallow / shoreline / shader / overlay。
```

因此最终选择先做整图 bake：

```text
地图 30x30，每格 64px -> 1920x1920。
stage1_water_surface.png 只在 W/D 水域有像素，其余 alpha=0。
```


当前状态：

- `stage1_shore_shadow.png` / `stage1_shore_shallow.png` 当前只是透明占位。
- `stage1_water_debug.png` 用于快速看水层和临时背景的合成。

### 4.1 失败尝试：单独 shallow / shadow 色带

最初尝试过：

```text
shore_shadow: 水陆边界附近深蓝暗边
shore_shallow: 水侧青蓝浅滩色带
```

问题：

- 在窄河里读成「两条淡色直线」。
- shallow 变成 cyan 发光边。
- shadow 变成岸边竖线。
- 不是《尤特娜》那种从河心到岸边的自然渐变。

结论：

```text
不要单独使用 shore_shadow / cyan shore_shallow overlay。
这种做法容易产生等值线和描边感。
```

### 4.2 失败尝试：distance field shallow layer

后来把 shallow 改成水内距离场：

```text
水域内部每个像素算离岸距离。
离岸近 -> 浅。
往河心 -> 深。
```

这比两条 band 更合理，但单独 debug 图里仍然容易看成线，尤其 alpha 边叠在临时绿底上会放大轮廓感。

### 4.3 当前方案：水面自身做 30px 内侧提亮

用户提出更直接的要求：

```text
边缘 30px 开始渐渐调亮。
```

于是改成：

```python
edge = np.clip(1.0 - dist / 30.0, 0.0, 1.0)
rgb *= (1.0 + edge * strength)
rgb += edge * additive
```

做过三档：

```text
1x:  rgb *= 1 + edge * 0.18, rgb += edge * 10
3x:  rgb *= 1 + edge * 0.54, rgb += edge * 30
1.5x: rgb *= 1 + edge * 0.27, rgb += edge * 15
```

当前保留 1.5x：

```python
rgb *= (1.0 + (edge * 0.27)[:, :, None])
rgb += (edge * 15.0)[:, :, None]
```

这个效果比较克制，不会像 3x 那样电光，但能在水边略微提亮。

## 六、GPT shoreline decal 实验

### 6.1 目标

用户认为「手绘 == AI 生图」，但关键不是让 AI 画整图，而是让 AI 生成可控岸线素材。

因此尝试：

```text

Python 抠图、切图、调色、沿水陆边界 bake。
```

### 6.2 GPT Image 透明 vs 灰底对比

生成：

```text
art_undecided/tiles/shoreline/gpt15_medium_shoreline_transparent.png
art_undecided/tiles/shoreline/gpt15_medium_shoreline_opaque_gray.png
art_undecided/tiles/shoreline/shoreline_gpt15_medium_transparent_vs_opaque.png
```

观察：

- 透明版确实有 RGBA alpha，但形状更像小块地形贴纸。
- 灰底版更适合作为抠图源。
- 颜色偏亮，不能原色贴进当前地图。

结论：选择灰底版作为素材源。

### 6.3 切 decal 与烘 shoreline

新增脚本：

```text
scripts/tiles/bake_stage1_shore_land.py
```

输出：

```text
art_undecided/tiles/shoreline/cut/decal_00.png ... decal_05.png
art_undecided/tiles/shoreline/cut/decal_contact.png
Assets/Art/Maps/stage1_shore_land.png
Assets/Art/Maps/stage1_shore_preview.png
Assets/Art/Maps/stage1_shore_preview_rgb.png
```

### 6.4 失败 1：逐边贴 strip

第一版沿每条水陆格边贴完整 strip。

结果：

```text
重复感很重。
像每个格子边都贴了一个小岛边框。
台阶状 / 贴纸感明显。
```

### 6.5 失败 2：连续 mask + AI noise

第二版改成：

```text
用水陆 mask 生成连续岸线带。
AI decal 只当纹理/noise 来源。
```

结果：

```text
重复感减少。
但 AI 手绘价值被稀释，只剩程序色带。
```

### 6.6 修复：straight alpha bug

中间发现一个重要 bug：

```text
stage1_shore_land.png 的 RGB 被写成接近 premultiplied alpha / 黑污染状态。
PNG/Unity 使用 straight alpha，透明边 RGB 如果接近黑，合成后会像脏阴影。
```

修复：

```python
straight_rgb[visible] = rgb[visible] / alpha[visible, None]
```

修完后透明边不再带黑绿污染。

### 6.7 结论

这条路线暂时不建议继续投入：

```text
GPT shoreline strip sheet -> 自动整条河铺满
```

效果不是最终想要的手绘岸线。更靠谱的 AI 使用方式应当是：

```text
Seedream 直接整体增强地图。
或者少量局部 decal placement，而不是整条河自动铺满。
```

区别：

### 8.2 改脚本支持多 ref

修改：

```text
scripts/volcengine_seedream_ref_portrait.py
```

原来：

```python
parser.add_argument("--ref", default=DEFAULT_REF)
"image": data_url(ref_path)
```

改为：

```python
parser.add_argument("--ref", action="append", default=None)
image_arg = data_url(ref_paths[0]) if len(ref_paths) == 1 else [data_url(path) for path in ref_paths]
"image": image_arg
```

保持向后兼容：

```text
单 ref -> image 字符串
多 ref -> image 数组
```

## 十四、关键结论

### 14.1 水面层

```text
GPT0 512 b90_s92 + 整图 bake 比旧 2x2 water tile 明显更强。
但单独调水参数不会解决整体地图手绘感。
```

### 14.2 程序 shoreline

```text
程序 mask / shallow / shadow / strip 自动铺满都不理想。
容易变成线、边框、色带、贴纸。
```

### 14.3 GPT shoreline decal

```text
GPT 可以给 brush 参考，但「6 个 strip 自动铺满整条河」路线不适合。
这批素材不建议继续作为主路线。
```

### 14.4 Seedream 整图增强

```text
Seedream 对地图布局保持能力比预期好。
单 ref 和多 ref 都能保住大结构。
多 ref 能明显引入 shoreline 风格参考。
```

### 14.5 当前最有前途路线

```text
用程序 / JSON 保持逻辑。
用 Seedream 生成整图地形视觉 overlay 或 patch source。
Unity 保留 objects / units / walkable 逻辑。
```

也就是说，比起继续局部调 tile，更现实的方向是：

```text
AI 整体增强地面层 + 程序对象层 + JSON 逻辑层
```

### 14.6 当前最可看的版本

```text
V3: 更干净、更自然，沙滩偏亮偏宽。
V5: 更手绘、不规则，边缘自然化更多，但偏脏、湿泥暗边重。
```

用户 Unity 截图中表示 V3/V5 overlay 路线「比原来好看」。

## 十五、后续建议

短期下一步：

1. 在 Unity 中继续比较 V3 / V5 overlay，决定哪张作为当前展示基准。
2. 如果继续 prompt：以 V3 为基准，小幅降低沙滩亮度和宽度，不要像 V4 那样过度克制。
3. 或以 V5 为基准，压掉深色湿泥边，让水道视觉不要被压窄。
4. 确认 overlay 是否应作为临时展示层，还是要拆成可复用 patch / macro texture。

中期路线：

```text
stage1_map.json 继续作为逻辑真源。
Seedream 输出作为视觉 ground overlay。
Objects（房子、树、桥、石头、单位）继续独立渲染。
```

需要注意：

```text
Seedream overlay 可能改动视觉边界。
如果视觉边界和逻辑格子差异太大，会影响玩家判断 walkable。
因此最终不能完全无约束使用 AI 整图，需要控制边界漂移幅度。
```
