## 三、Unity 加载 map_v0：踩的坑

### 3.1 切图路径要可配置

`MapSceneBuilder.cs` 原本把 `demo_map` 路径硬编码。为了能切版本，先尝试用 C# 插值字符串常量：

```csharp
private const string MapName = "map_v0";
private const string MapPng = $"Assets/Art/Maps/{MapName}_render.png";
```

**报错**：`CS8773: Feature 'constant interpolated strings' is not available in C# 9.0`。项目是 C# 9.0，常量插值字符串要 C# 11+。

**修复**：改成传统字符串拼接 `"Assets/Art/Maps/" + MapName + "_render.png"`。后来为简单起见直接写死成 `map_v0` 两个常量路径。教训：**Unity 这套 C# 版本偏旧，少用新语法糖**。

### 3.2 场景里旧物体没清，加载的还是老地图

改完路径、点了菜单生成，结果场景里**还是 demo_map**。原因是 Hierarchy 里已经有上一次生成的 `MapGrid` / `MapBackground` / `LingShuang`，新的又叠了一份，MapGrid 组件指着旧的 `demo_map`。

**修复**：除了 `Main Camera`，把场景里其它对象全删，再点 `Tools → FantacyCentry → Build Demo Map Scene` 重新生成。这次 Console 正确打印 `Placed LingShuang at cell (17, 17) on a 35x35 grid`，地图正确切到 map_v0。

教训：这个 builder 是“叠加生成”不是“清场重建”，**重建前要手动清场**（除 Main Camera）。

### 3.3 验证结果

Play 实测：可以走、有碰撞、walkable 正确（森林/水/悬崖/建筑不可走，草/路/桥可走，桥头开的那条通道也能下来）。**第一版地图逻辑跑通。** 美术质感留到最后再换。

## 四、程序化 tile 提质 + 与 AI 出图的边界（重要认知）

### 4.1 做了什么

直接在 `gen_base_tiles.py` 里把各 tile 的画法升级了一版（不改接口/输出名）：
- 草地：加明暗簇、花点、泥点，减少平铺感。
- 水面：加浪峰高光，更有流动感。
- 路面：石砖破损点更自然。
- 桥：木板纹理 + 侧梁 + 铆钉。
- 岩石/悬崖：明暗 + 裂纹更明确。
- 森林：冠层深浅更丰富。
- 房子：屋顶分层瓦线 + 门窗细节。

重新生成 + 重渲染整图，确认仍然无缝。

### 4.2 关键认知：程序化 ≠ AI 出图

用户问“能不能画得像 GPT-Image 那种质感”。**老实结论：程序化脚本做不到**。

- 程序化 tile 的定位是：**可控、无缝、能跑通的技术底图（graybox 级别）**，不会有 AI 绘画那种笔触和质感。
- 想要 GPT-Image 风格 → 必须用 **AI 出图**（GPT/Gemini，按现成 prompt），我只能做**后处理**（抠图、无缝检查、统一色阶、切片）和**筛选质检**。
- Aseprite 手工像素润色这一步，**我也不能单干**，需要人或美术操作。

而且这两份 prompt 文档**早就写好了**（`docs/prompts/ui-prompts.md`、`docs/prompts/tile-prompts.md`），不用重复造。一度我又重复建议“写新 prompt”，被用户纠正——已经有了。

### 4.3 路线确认（不要跑偏）

明确的开发顺序：**① 先用现在这张丑图（map_v0）验证战斗逻辑 + UI → ② 最后再搞地图美术（AI 出图 + Aseprite）**。所以现在不纠结地图好不好看，往战斗推进。

## 五、UI prompt 文档 + 程序化范围格（红蓝格子 / 选择框）

### 5.1 两份 prompt 文档

在 `docs/prompts/` 下建了并打磨了两份提示词文档：
- `ui-prompts.md` —— UI 套件出图工作流（concept sheet 定调 → 作为 init 图 → 逐件细化 → Python 抠图 → Unity 9-slice），锁定了风格（深蓝半透明面板 + 金色花边），并给了每个 UI 件（主面板 / 三态按钮 / HP·MP 条 / 指令图标 / 选择框 / 单位卡 / 阶段横幅）的独立 prompt。
- `tile-prompts.md` —— tile 出图提示词（草/路/水/桥/田 + 边界 tile + Flux 变体 + 脚本后处理 + 检查清单）。

关键经验记进文档：**UI 出图 Gemini（Nano Banana）比 GPT-Image 好**（GPT 容易烤进假文字、3D 高光）；要某一件时 prompt 必须写“ONLY … NOTHING ELSE / 不要画别的”，否则模型会重画整套。

### 5.2 范围格为什么必须用脚本而不是 AI

写了 `scripts/ui/gen_range_tiles.py`，程序化生成 SRPG 的范围叠加格和选择框（真 alpha）。

**为什么不能让 AI 画**（这点很关键，记下来）：AI 出图**做不出真正的半透明 alpha** —— 它只会画一个更浅的纯色块、加噪点、有时还烤个水印。而范围格就是**纯几何 + 真透明**，所以必须确定性地用脚本生成。

输出到 `Assets/Art/UI/range/`（64×64，真 alpha）：
- `move_tile.png` —— 半透明**蓝色**移动范围格，1px 深色边
- `attack_tile.png` —— 半透明**红色**攻击范围格，1px 深色边
- `select_frame.png` —— 金色四角括号选择框，中心完全透明

教训沉淀成一条规则：**alpha 叠加 + 纯几何 → 用脚本，不要用 AI**。

## 六、五角色 rig 一键装配

### 6.1 角色搬进 Assets 的目录规则

源在 `art/characters/<角色>/<角色>/` 是**双层**的（里面才是 `animations/` + `rotations/`），进 Unity 要**拍平一层**到 `Assets/Art/Characters/<角色>/`，和 LingShuang 保持一致。不要套成 `EmpireArcher/EmpireArcher/`。

用户把 EmpireAxeSoldier / LuLi / SuYao 也搬了进来，加上原有的 EmpireArcher、LingShuang，一共 **5 个角色**。

### 6.2 统一动作文件夹命名（关键决定）

PixelLab **V3 用直接 prompt 而不是模板**，所以生成的动作文件夹名又长又乱（如 `Archer_draws_bow_and_releases_an_arrow_in_a_clean`）。

决定：**每个角色统一成 4 个标准动作文件夹** —— `walk` / `idle` / `attack` / `react`（对应游戏 4 动作）。这样装配脚本里那一大堆关键词匹配可以彻底删掉。

四角色实际映射（核对无误，每个正好 4 个文件夹）：

| 角色 | walk | idle | attack | react |
|---|---|---|---|---|
| EmpireArcher | marches_forward | stands_in_idle | draws_bow | takes_a_sharp_hit |
| EmpireAxeSoldier | marches_forward | stands_in_idle | swings_battle_axe | takes_a_hit_braces |
| LuLi | walks_forward | stands_in_idle | draws_longsword | takes_a_hit_recoils |
| SuYao | walks_forward | stands_in_idle | casts_spell | takes_a_hit_recoils |
| LingShuang | walks_forward | stands_in_idle | forward_thrust | takes_a_hit_flinches |

### 6.3 批量重命名踩的坑（重要）

写了个 PowerShell 脚本按关键词把长文件夹名 Move 成 `walk/idle/attack/react`（连 `.meta` 一起移，保 GUID）。

**坑 1**：长命令在终端里**显示被截断**，我以为没执行完，又**重跑了一次**。结果第一次其实已经成功（文件夹已经是 walk/idle/attack/react），第二次又按旧逻辑跑，把已经叫 `idle`/`react` 的文件夹误判成 attack，**移进了 `attack/idle`、`attack/react` 子目录**，结构被搞乱。

**坑 2**：分类逻辑里 `idle`/`react` 没被精确匹配，落到了 `else → attack` 分支，所以才会被塞进 attack 下。

**修复**：先实地 `Get-ChildItem -Recurse` 看清真实结构（确认 sprite 的 png/.meta 没动、GUID 安全），再把 `attack/idle`、`attack/react` 移回顶层。最终每个角色干净的 4 个顶层文件夹，`badNested=0`。

教训：
- **批处理移动文件前先 dry-run / 先看结构**，别盲跑。
- 终端**显示截断 ≠ 没执行**，不要想当然重跑有副作用的命令。

### 6.4 装配脚本改造

`CharacterRigBuilder.cs` 做了两处改动：
1. **`TryClassify` 简化**：从一大堆 `Contains("bow")/("axe")/("march")…` 关键词匹配，改成精确 `switch`：`walk→Walk`、`idle→Idle`、`attack→Attack`、`react→Hit`。
2. **新增菜单 `Tools → FantacyCentry → Build All Character Rigs`**：扫描 `Assets/Art/Characters/` 下所有角色，逐个生成 prefab（不用一个个点）。

### 6.5 结果

一键 Build All 成功，5 个角色各 **12 clips**（4 动作 × 3 朝向 S/E/N），无分类失败。prefab 各自存到 `Assets/Art/Characters/<角色>/<角色>.prefab`。

## 七、发现的下一个问题：角色一起动

把多个角色拖进场景 Play，发现**一按方向键所有人一起走**。

原因：每个 prefab 都挂了 `CharacterDemoController`，这个脚本在各自的 `Update()` 里**直接读箭头键并自己移动**。5 个角色 = 5 个监听器，所以齐步走。这是 demo 测试脚本的预期行为（本来给单角色试走地图用），不是 bug。

**结论 / 下一步**：该废弃 `CharacterDemoController` 的“箭头直接走”，进入真正的 SRPG 选择制——**点击选中单位 → 显示可移动范围 → 点目标格才移动**，每个单位独立、按回合走。
